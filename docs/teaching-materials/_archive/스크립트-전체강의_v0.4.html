명확하게 말씀드릴게요. 이 강의는 처음부터 끝까지 한 땀 한 땀 같이 만들어가는 강의가 아닙니다. 그거 몇 달 걸려요. 저 이미 만들어 왔습니다.

이 시간을 어떻게 쓰면 좋을까 많이 고민했어요. 결론은 이겁니다. 여러분이 오늘 집에 가서 혼자 써먹을 수 있는 것들을 드리자. 소스코드 다 드립니다. 집에서 혼자 분석하는 법도 알려드립니다.

AI 도구 처음 켜기

⏱ 20분 💡 도구 소개 + 가능성 각인
Slide 2-1
Claude Code란 무엇인가

Claude Code는 Anthropic이 만든 AI 코딩 도구입니다. ChatGPT와 비슷하지 않냐고 하시는 분들이 있는데, 결정적인 차이가 있습니다. ChatGPT는 대화창에서 코드를 보여줍니다. Claude Code는 실제로 파일을 열고, 수정하고, 저장하고, 실행합니다.

쉽게 말씀드리면, ChatGPT는 레시피를 알려주는 요리책이고, Claude Code는 실제로 주방에서 요리를 해주는 셰프입니다. 여러분은 "오늘 저녁 뭘 먹을까요?"라고 말씀만 하시면 됩니다.

월 $20, 한화 약 3만원입니다. 이 금액으로 전문 소프트웨어 엔지니어를 24시간 고용하는 것과 비슷한 효과를 얻을 수 있습니다. 오늘 강의에서 제가 보여드릴 us-stock 시스템도 100% Claude Code로 만들었습니다.

💬 청중 참여
"혹시 Claude Code 써보신 분 계십니까? — 안 써보셨다면 오늘 이 강의가 첫 번째 경험이 될 겁니다. 써보셨다면 오늘 좀 더 깊은 활용법을 가져가시면 됩니다."
그렇다면 기존 방식과 비교해서 뭐가 달라지는지 구체적으로 보여드리겠습니다.
Slide 2-2
기존 개발 방식 vs Claude Code

예전에는 주식 분석 프로그램을 하나 만들려면 Python을 배워야 했고, 라이브러리 문서를 읽어야 했고, Stack Overflow를 뒤지면서 에러와 씨름해야 했습니다. 보통 사람에게는 진입장벽이 너무 높았죠.

Claude Code를 쓰면 이렇게 합니다. "S&P 500 500개 종목의 지난 1년 주가 데이터를 가져와서 RSI가 30 이하인 종목만 뽑아줘" — 이렇게 한국어로 말합니다. 그러면 Claude Code가 Python 코드를 짜고, 실행하고, 결과를 보여줍니다. 에러가 나면 스스로 고칩니다.

실제로 저는 이 시스템의 핵심 분석 엔진을 사흘 만에 만들었습니다. 혼자 코딩했다면 몇 달이 걸렸을 내용입니다. 코딩 실력이 없어도 됩니다. 무엇을 원하는지 명확히 말할 수 있으면 됩니다.

그런데 Claude Code만 가지고는 뭔가 빠집니다. 지식을 쌓고 정리하는 도구도 필요합니다. 그게 바로 Obsidian입니다.
Slide 2-3
Obsidian — 나만의 두 번째 뇌

Obsidian은 노트 앱입니다. 그런데 일반 메모장이나 노션과 다른 점이 있습니다. 메모와 메모 사이의 연결 관계를 시각화해줍니다. 마치 뇌의 신경망처럼요.

저는 투자 관련 공부를 하면서 나오는 모든 내용을 Obsidian에 기록합니다. "금리가 오르면 채권 가격이 내린다"는 내용을 기록하면, 그 노트가 "연방준비제도", "채권 ETF", "포트폴리오 전략"과 자동으로 연결됩니다. 나중에 비슷한 주제를 공부할 때 그 연결이 빛을 발합니다.

오늘 이 강의 자료도 Obsidian으로 관리하고 있습니다. 그리고 이 강의 스크립트 자체도 Claude Code가 Obsidian에 있는 내용을 읽어서 만들었습니다. AI와 지식관리 도구가 결합되는 순간, 개인의 생산성이 기하급수적으로 올라갑니다.

두 도구를 어떻게 실제로 연결해서 쓰는지, 제 실제 작업 흐름을 보여드리겠습니다.
Slide 2-4
실제 작업 플로우 — 강의 준비부터 분석까지

제가 이 강의를 준비한 과정을 말씀드리겠습니다. 먼저 Obsidian에 강의에서 다룰 주제들을 노트로 정리했습니다. 거시경제, 밸류에이션, 기술적 분석, 시스템 구조 — 각각 노트로 만들어서 연결했습니다.

그 다음 Claude Code에게 말했습니다. "이 노트들을 읽어서 150분짜리 강의 스크립트를 작성해줘. 합쇼체로, 청중 참여 포인트도 넣어줘." Claude Code가 파일을 읽고, 내용을 분석하고, 지금 여러분이 보고 계신 이 스크립트를 만들었습니다.

강의 준비 시간이 보통 2주 걸리던 것이 2~3일로 줄었습니다. 그리고 내용은 훨씬 더 체계적이 되었습니다. 이것이 AI를 활용한 지식 노동의 미래입니다.

💬 청중 참여
"여러분은 공부한 내용을 어디에 기록하십니까? 노트앱? 엑셀? 아니면 머릿속에만 저장하십니까? — 머릿속에 저장하신 분은 오늘부터 Obsidian을 시작해보시기 바랍니다."
그럼 Claude Code를 어떻게 시작하면 되는지, 실제 설치와 비용을 말씀드리겠습니다.
Slide 2-5
설치 및 시작 방법 — 오늘 바로 시작할 수 있습니다

Claude Code 설치는 터미널 한 줄이면 됩니다. npm install -g @anthropic-ai/claude-code — 이 명령어 하나입니다. Anthropic 홈페이지에서 API 키를 발급받고 연결하면 바로 쓸 수 있습니다.

비용 구조를 말씀드리면, Claude Code 구독료는 월 $20입니다. 한화로 약 3만원이죠. 여기에 API 사용료가 추가됩니다. 오늘 보여드릴 주식 분석 시스템을 하루 한 번 돌리면 하루 API 비용이 약 7원에서 42원입니다. 한 달에 1,000~1,300원 수준입니다.

Obsidian은 무료입니다. 기본 기능만으로도 충분합니다. 오늘 강의 마치시고 두 개 다 설치해 보시기를 강력 추천드립니다.

좋습니다. 도구를 알았으니 이제 '무엇에 쓸 것인가'로 넘어갑니다. 2026년 지금, 어떤 곳에 투자해야 할까요?
Slide 2-6
에이전트 팀 — 7+1명의 AI 동료

제가 운영하는 us-stock 시스템에는 Claude Code 에이전트 7명이 팀으로 일하고 있습니다. 각자 역할이 다릅니다.

팀장 격인 lead 에이전트가 전체 일정을 조율하고, quant-analyst는 수치 분석 로직을 담당합니다. data-engineer는 데이터 수집 파이프라인을 안정적으로 유지하고, ai-strategist는 AI 분석의 품질과 환각 방지를 담당합니다. dashboard-dev는 여러분이 보실 대시보드 화면을 만들고, qa는 품질 검증을 합니다. 마지막으로 service-evolver라는 에이전트는 매일 시스템을 스스로 1가지씩 개선합니다.

이 에이전트들이 매일 새벽 4~5시간 동안 일하고, 아침 7시에 결과를 내놓습니다. 저는 아침에 일어나서 5분 동안 결과만 확인하면 됩니다. 이것이 AI 시대의 개인 투자자의 모습입니다.

이제 투자 환경 자체를 봐야 할 시간입니다. 2026년 지금, 시장은 어떤 상황입니까?

Slide 3 · IT-1
IT 개념 5분 정리 — 왜 알아야 하는가

여러분, 잠깐 저한테 5분만 주십시오. 이 5분 동안 IT 개발자들이 매일 쓰는 핵심 개념들을 설명해 드리겠습니다. 코딩을 모르셔도 됩니다. 개념만 알면 충분합니다.

왜 IT 개념을 주식 투자 강의에서 다루느냐고 하시겠죠. 이유는 간단합니다. 2026년 현재 가장 돈이 몰리는 곳이 AI 인프라입니다. 엔비디아, AMD, TSMC, AWS, Azure — 이 기업들이 무슨 사업을 하는지 구조적으로 이해하면 투자 판단이 달라집니다.

지금부터 설명드릴 내용은 OS, CPU, 메모리, 웹, 클라우드, 컨테이너입니다. 어렵지 않습니다. 오히려 이걸 알고 나면 '아, 그래서 엔비디아가 그렇게 비싸구나'가 이해됩니다.

가장 기본부터 시작합니다. 컴퓨터를 켜면 제일 먼저 실행되는 것, OS입니다.
Slide 3 · IT-2
OS 커널 구조 — 하드웨어의 문지기

컴퓨터를 켜면 가장 먼저 실행되는 것이 운영체제, 즉 OS입니다. OS는 두 부분으로 나뉩니다. 커널과 사용자 공간입니다.

커널은 운영체제의 핵심입니다. CPU, 메모리, 디스크에 직접 접근합니다. 우리가 쓰는 앱들은 커널 위에서 실행됩니다. 앱이 파일을 읽거나 네트워크에 연결할 때는 커널에게 요청합니다. 이 요청을 시스템 콜이라고 합니다.

비유하자면 커널은 건물 관리자입니다. 우리는 각 층에 살고 있고, 전기를 쓰거나 물을 쓰려면 관리자에게 요청해야 합니다. 직접 전기실에 들어갈 수 없는 것처럼, 앱이 직접 하드웨어를 만질 수 없습니다. 커널이 그 역할을 대신합니다.

그러면 그 하드웨어 중 핵심인 CPU와 메모리를 보겠습니다.
Slide 3 · IT-3
CPU + 메모리 구조 — AI 시대에 GPU가 왜 비싼가

CPU는 연산을 담당합니다. 제어장치는 '무엇을 할지' 결정하고, 연산장치는 실제 계산을 합니다. CPU 안에는 레지스터라는 초고속 임시 저장소가 있습니다.

메모리는 계층 구조입니다. CPU 레지스터가 가장 빠릅니다. 그 다음 캐시 메모리, RAM, SSD/HDD 순입니다. 빠를수록 용량이 작고 비쌉니다. 느릴수록 대용량이고 저렴합니다.

이게 왜 투자에 중요합니까? AI 학습은 엄청난 양의 행렬 연산을 합니다. CPU는 직렬 처리, GPU는 병렬 처리에 특화되어 있습니다. GPU 코어 수가 CPU보다 수십 배 많습니다. 이 차이가 AI 시대에 엔비디아가 독주하는 이유입니다. H100 하나가 3만 달러인 이유입니다.

이제 컴퓨터가 여러 프로그램을 동시에 실행하는 원리를 볼 차례입니다.
Slide 3 · IT-4 ⭐
Context Switching — 멀티태스킹의 비밀

컴퓨터는 여러 프로그램을 '동시에' 실행합니다. 지금 이 강의를 들으시면서 음악도 틀고 메모도 하시는 분 계십니까? 이게 어떻게 가능할까요?

비밀은 Context Switching입니다. CPU는 사실 한 번에 하나만 처리합니다. 그런데 너무 빠르게 전환하기 때문에 동시에 하는 것처럼 보입니다. 1초에 수천 번씩 '지금은 크롬 처리, 지금은 카카오톡 처리, 지금은 음악 앱 처리'를 반복합니다.

이때 PCB가 중요합니다. PCB는 프로세스의 현재 상태를 저장해 두는 서류철입니다. '어디까지 했는지, 레지스터 값이 뭔지' 저장해 두고, 나중에 돌아올 때 그걸 보고 이어서 실행합니다. Context Switching은 멀티태스킹의 핵심 원리입니다. 이 개념이 뒤에 나올 프로세스, 스레드 개념의 기반입니다.

💬 청중 참여
"지금 이 순간 여러분 스마트폰에 몇 개의 앱이 실행 중입니까? 한번 확인해 보십시오. 10개? 20개? — 이 모든 것을 CPU 하나가 처리합니다. Context Switching 덕분입니다."
프로세스가 어떤 상태들을 거치는지, 상태 머신으로 이해해 보겠습니다.
Slide 3 · IT-5
프로세스 상태 머신 — NEW에서 TERMINATED까지

프로세스가 거치는 상태는 총 다섯 가지입니다. 프로그램을 실행하면 NEW 상태로 생성됩니다. 그 다음 READY 상태가 됩니다. CPU를 받을 준비가 된 상태입니다. CPU 스케줄러가 선택하면 RUNNING 상태가 됩니다. 실제로 CPU에서 실행 중인 상태입니다.

실행 중에 파일을 읽거나 네트워크 응답을 기다려야 한다면 WAITING 상태로 전환됩니다. I/O 작업이 완료되면 다시 READY로 돌아옵니다. 프로그램이 종료되면 TERMINATED 상태가 됩니다.

왜 WAITING 상태가 따로 필요할까요? CPU는 나노초 단위로 작동하지만 I/O 디스크는 밀리초 단위입니다. 10만 배 차이입니다. 기다리는 동안 CPU를 붙잡고 있으면 낭비입니다. WAITING으로 빼고 다른 프로세스에게 CPU를 넘겨주는 것이 훨씬 효율적입니다. 이것이 멀티태스킹의 실제 작동 방식입니다.

프로세스 안에서 더 작은 실행 단위인 Thread를 살펴보겠습니다.
Slide 3 · IT-6
Thread vs Process — 메모리 공유가 핵심 차이

Process는 독립적인 실행 단위입니다. 각자 자신만의 메모리 공간을 가집니다. Thread는 Process 안에서 실행되는 더 작은 단위입니다. 차이는 메모리 공유 여부입니다.

같은 Process 안의 Thread들은 CODE, DATA, HEAP 메모리를 함께 씁니다. 단, STACK은 Thread마다 개별적으로 가집니다. 이 공유 덕분에 Thread 간 통신이 빠릅니다. 반면 별개의 Process들은 메모리를 공유하지 않아 완전히 독립됩니다.

실제 예를 들면, Chrome은 탭마다 독립 Process를 씁니다. 탭 하나가 멈춰도 다른 탭은 살아있습니다. Firefox는 4개 Process와 다수 Thread를 씁니다. 메모리를 더 효율적으로 씁니다. 두 방식 모두 장단점이 있고, 설계 상황에 따라 선택합니다.

이제 웹 통신의 기초로 넘어가겠습니다. URL과 HTTP입니다.
Slide 3 · IT-7
WEB 기초 — URL · HTTP · 상태코드

웹의 기초를 설명드리겠습니다. URL은 '어디에 있는 자원인지'를 표시합니다. https://example.com/path 에서 https는 프로토콜, example.com은 서버 주소, /path는 자원의 위치입니다.

HTTP는 요청과 응답의 규약입니다. 요청할 때 Method가 있습니다. GET은 데이터 조회, POST는 데이터 전송, PUT은 수정, DELETE는 삭제입니다. 응답에는 상태 코드가 있습니다. 200은 성공, 404는 '없음', 500은 서버 오류입니다.

여러분이 유튜브에서 영상을 클릭하는 것, 쇼핑몰에서 주문하는 것, 뱅킹 앱을 쓰는 것 — 모두 이 HTTP 요청/응답으로 작동합니다. 눈에 보이지 않지만 초당 수백만 번씩 일어나고 있습니다.

이 웹 요청을 받아서 처리하는 서버 구조, 3-Tier를 보겠습니다.
Slide 3 · IT-8
웹서비스 3-Tier 아키텍처 — 프론트·백엔드·DB

웹 서비스는 크게 세 층으로 나뉩니다. 첫 번째 층은 프론트엔드입니다. 여러분의 브라우저에서 실행됩니다. 화면을 그리고 버튼 클릭을 처리합니다. HTML, CSS, JavaScript로 만들어집니다.

두 번째 층은 백엔드입니다. 서버에서 실행됩니다. 프론트엔드의 요청을 받아서 비즈니스 로직을 처리합니다. '이 사용자가 로그인됐는지', '이 주문이 유효한지'를 판단합니다.

세 번째 층은 데이터베이스입니다. 데이터를 영구적으로 저장합니다. 여러분의 계정 정보, 주문 내역, 게시물이 모두 여기에 있습니다. us-stock 시스템도 이 구조입니다. Next.js 대시보드가 프론트엔드, Python 분석 서버가 백엔드, SQLite가 데이터베이스입니다.

이 층들 사이에서 데이터를 주고받는 규칙이 API입니다.
Slide 3 · IT-9
API & REST API — 두 프로그램이 대화하는 방법

API는 Application Programming Interface의 약자입니다. 두 프로그램이 서로 대화하는 방법입니다. 비유하면 식당의 메뉴판입니다. '이 메뉴를 이 방법으로 주문하면 이렇게 가져다 드립니다'라는 약속입니다.

REST API는 웹에서 가장 널리 쓰이는 API 설계 방식입니다. URI로 자원을 표현하고, HTTP Method로 행위를 표현합니다. GET /stocks/AAPL 은 'AAPL 종목 정보를 조회해줘'라는 뜻입니다. 응답은 JSON 형식으로 옵니다.

us-stock 시스템에서는 API가 세 곳에 씁니다. 프론트엔드가 백엔드에게 데이터를 요청할 때, Yahoo Finance에서 주가를 받아올 때, Gemini에게 AI 분석을 요청할 때입니다. 여러분이 Claude Code에게 '데이터 가져와줘'라고 하면 이 API들이 뒤에서 작동합니다.

마지막 IT 개념입니다. 클라우드 시대의 핵심 기술, Docker입니다.
Slide 3 · IT-10
Docker & Container — 밀키트로 이해하는 컨테이너

Docker는 애플리케이션을 컨테이너에 담아서 실행하는 기술입니다. Image는 불변의 설계도입니다. 한 번 만들면 바뀌지 않습니다. Container는 그 Image를 실행한 인스턴스입니다. 같은 Image에서 Container를 수십 개 만들 수 있습니다.

VM, 즉 가상머신과 비교하면 Docker가 훨씬 가볍습니다. VM은 OS 전체를 가상화해서 수 GB가 넘습니다. Docker Container는 프로세스 수준 격리로 수십 MB입니다. 시작 시간도 수 분 대 수 초입니다. 그래서 AWS, Azure, GCP의 모든 클라우드 서비스가 Docker 기반으로 돌아갑니다.

우리의 us-stock 대시보드가 Synology NAS에서 Docker Container로 실행됩니다. 그래서 어떤 환경에서도 동일하게 작동합니다. 나중에 5부에서 직접 보여드리겠습니다.

IT 개념 10분 완료입니다. 이제 이 개념들이 AI 투자 흐름과 어떻게 연결되는지 보겠습니다.

그런데 AI가 이 운동장을 조금씩 평평하게 만들고 있습니다. 예전에는 기관 투자자만 가질 수 있었던 503개 종목 전체 스크리닝 기능을, 이제 개인도 월 $20짜리 Claude Code로 만들 수 있습니다. 예전에는 금융 데이터 분석에 전문가가 필요했지만, 이제는 한국어로 말하면 됩니다.

Slide 3 · REF-1
네트워크 핵심 — L2·L3·L4 · Encapsulation · TCP vs UDP

네트워크 계층별 식별자를 정리합니다. L2 계층에서는 MAC 주소로 NIC, 즉 네트워크 카드를 식별합니다. 48비트 물리 주소이며 같은 네트워크 안에서 통신할 때 씁니다. L3 계층에서는 IP 주소로 호스트, 즉 컴퓨터를 식별합니다. 라우터가 목적지를 찾을 때 씁니다. L4 계층에서는 포트 번호로 서비스를 식별합니다. HTTP는 80번, HTTPS는 443번, SSH는 22번입니다.

Encapsulation은 데이터가 전송될 때 각 계층이 헤더를 추가하는 과정입니다. Stream에서 시작해서 L4가 TCP/UDP 헤더를 붙여 Segment, L3가 IP 헤더를 붙여 Packet, L2가 MAC 헤더를 붙여 Frame이 됩니다. 수신 쪽에서는 반대로 헤더를 벗겨냅니다.

TCP는 연결 지향 프로토콜입니다. 3-way handshake로 연결을 맺고 순서를 보장합니다. HTTP, 파일 전송에 씁니다. UDP는 비연결 프로토콜입니다. 보장 없이 빠르게 전송합니다. DNS 조회, 스트리밍, 게임에 씁니다. 신뢰성이 필요하면 TCP, 속도가 우선이면 UDP입니다.

OS 심화 — 메모리 레이아웃 · Compile 과정 · 프로세스 생성

프로세스의 메모리는 네 영역으로 나뉩니다. CODE 영역은 실행 코드를 저장합니다. 읽기 전용입니다. DATA 영역은 전역 변수와 static 변수를 저장합니다. HEAP은 malloc 같은 동적 할당 메모리이며 위 방향으로 자랍니다. STACK은 함수 호출과 지역 변수이며 아래 방향으로 자랍니다. HEAP과 STACK이 서로를 향해 자라다가 만나면 메모리 오류가 납니다.

Compile 과정은 5단계입니다. .c 소스파일에서 시작해 Preprocessor가 #include와 #define을 처리하고, Compiler가 어셈블리로 변환하고, Assembler가 오브젝트 파일을 만들고, Linker가 라이브러리와 결합해서 최종 실행 파일이 됩니다. Python이나 JavaScript는 인터프리터 방식으로 이 과정 없이 바로 실행합니다.

프로세스 생성은 fork()와 exec() 두 단계입니다. fork()는 현재 프로세스를 복사해서 자식 프로세스를 만듭니다. exec()는 자식 프로세스를 새 프로그램으로 교체합니다. 여러분이 터미널에서 python을 실행할 때 이 과정이 일어납니다. 모든 프로세스는 이 방식으로 생성됩니다.

S&P 500이란?

S&P 500은 미국 주식 시장을 대표하는 지수입니다. 503개 종목으로 구성되어 있고, 전 세계 시가총액의 약 45%를 차지합니다. 코스피 전체가 전 세계의 2~3%인 것과 비교하면 규모가 다릅니다. 코스피가 아니라 삼성·SK·현대 최상위 503개를 모은 압축판 미국 경제라고 생각하시면 됩니다.

종목 비중은 시가총액 가중 방식입니다. 애플·마이크로소프트·엔비디아처럼 큰 기업이 더 높은 비중을 차지합니다. SPY ETF 하나를 사면 이 503개 종목을 한 번에 보유하는 것과 같습니다. 지난 10년 수익률이 약 250%로, 같은 기간 코스피는 거의 제자리였습니다.

미국 주식의 또 다른 장점은 달러 자산이라는 점입니다. 원화는 경제 위기가 오면 약세가 되지만 달러는 반대입니다. 위기가 올수록 안전자산 달러가 강세가 됩니다. 주가 상승 수익과 달러 헤지 효과를 동시에 누릴 수 있습니다.

시장 신호 3대장

시장 상태를 읽는 핵심 지표 3가지를 먼저 알아두십시오. 첫 번째는 VIX, 공포지수입니다. 20 이하면 시장이 평온한 상태, 30 이상이면 공포 상태, 80 이상이면 패닉입니다. 2020년 코로나 때 VIX가 85까지 올라갔습니다. VIX가 급등하면 기관이 매도하고 있다는 신호입니다.

두 번째는 Yield Curve, 장단기 금리차입니다. 10년물 국채 금리에서 3개월물 금리를 뺀 값인데, 이 값이 마이너스로 역전되면 역사적으로 12~18개월 후에 경기침체가 왔습니다. 예외가 없었습니다. 세 번째는 HY Spread, 하이일드 스프레드입니다. HYG/IEF 비율로 측정하는데, 이 비율이 벌어지면 기업 부도 위험이 높아지고 있다는 신호입니다.

이 3가지가 바로 Section 5에서 배울 5센서 중 3개와 직결됩니다. VIX 센서(30%), Yield Curve 센서(12%), Credit 센서(15%)가 각각 이 지표를 사용합니다. 오늘 개념을 익혀 두시면 Section 5가 훨씬 쉽게 느껴지실 겁니다.

이 지표들이 나빠지면 us-stock이 자동으로 시장 체제를 바꿉니다.
Slide 4-3
ETF 지도

us-stock 시스템에서 사용하는 ETF들을 먼저 알아두겠습니다. SPY는 S&P 500을 추종하며 10년 수익률 약 250%입니다. QQQ는 나스닥100을 추종하며 10년 약 450%로 더 공격적입니다. RSP는 동일가중 S&P 500 ETF인데, 503개 종목에 똑같은 비중을 줍니다. 시장 폭(Breadth)을 측정하는 센서로 활용됩니다.

HYG는 하이일드 채권 ETF, IEF는 중기 국채 ETF입니다. 이 두 개의 비율이 Credit 센서가 됩니다. 그리고 섹터 ETF 11개가 있습니다. XLK(기술), XLF(금융), XLE(에너지), XLV(헬스케어), XLI(산업재) 등 11개가 S&P 500의 11개 섹터를 각각 대표합니다.

11개 섹터 ETF는 야구팀 9개 포지션처럼 생각하시면 됩니다. 어떤 포지션이 강하고 약한지 보면 지금 시장이 어떤 테마를 선호하는지 알 수 있습니다. 섹터 히트맵에서 초록이 많으면 시장이 건강하고, 빨간 섹터가 많으면 위험 신호입니다.

시장 체제란?

"오늘 주식 투자하기 좋은 날인가?" 이 질문에 감으로 대답하지 말고 숫자로 대답하는 것이 시장 체제 감지입니다. us-stock은 5센서 점수를 합산해서 4단계 체제를 판정합니다. risk_on(0.75 미만), neutral(0.75~1.50), risk_off(1.50~2.25), crisis(2.25 이상)입니다.

기상청의 날씨 예보처럼 생각하시면 됩니다. 맑음(risk_on), 흐림(neutral), 비(risk_off), 태풍(crisis) 4단계입니다. 태풍 예보가 떴는데 소풍을 가면 안 되듯, crisis 체제에서 신규 매수를 늘리는 것은 위험합니다. 시스템이 체제를 판정하면 각 체제에 맞는 파라미터가 자동으로 적용됩니다.

중요한 것은 체제가 언제든 바뀔 수 있다는 점입니다. 어제 risk_on이었다고 내일도 그럴 거라는 보장이 없습니다. 그래서 매일 아침 5분 루틴으로 체제를 확인하는 습관이 필요합니다. Section 5에서 5센서가 어떻게 이 체제를 판단하는지 상세히 보겠습니다.

체제별로 손절 기준이 자동으로 바뀌는 것도 Section 5에서 확인합니다.
Slide 4-5
이동평균선

이동평균선은 기술적 분석의 가장 기본 도구입니다. 5일, 20일, 60일, 120일, 200일 평균 주가를 선으로 연결한 것입니다. 단기선이 장기선 위에 차례로 쌓여 있는 상태를 정배열이라고 하며, 강한 상승 추세를 의미합니다. 5 > 20 > 60 > 120일 순서로 위에 있으면 정배열입니다.

골든크로스는 단기 이동평균선이 장기 이동평균선을 아래에서 위로 돌파하는 순간입니다. 대표적으로 50일선이 200일선을 위로 넘는 것이 골든크로스이며, 많은 시스템의 매수 신호가 됩니다. 반대로 위에서 아래로 내려오는 것이 데드크로스, 매도 신호입니다. us-stock은 SMA50과 SMA200을 Trend 센서에 활용합니다.

200일선은 지난 1년치 주가의 평균입니다. 주가가 200일선 위에 있으면 작년보다 비싸게 사는 것이고, 아래에 있으면 싸게 사는 것입니다. 워런 버핏이 "200일선 위에서만 사라"고 한 것이 이 이유입니다. 간단하지만 강력한 필터입니다.

Slide 4-6
RSI · MACD · 볼린저밴드

RSI는 0에서 100 사이 숫자로 과매수·과매도를 측정합니다. 70 이상이면 과매수(조정 가능성), 30 이하면 과매도(반등 가능성)입니다. us-stock에서 RSI가 30 미만이면 Reversal(반전) 전략으로 분류되며, 낙폭 과대 종목을 역발상으로 포착하는 데 씁니다.

MACD는 두 개의 이동평균 차이를 이용해서 추세 전환을 감지합니다. MACD 선이 시그널 선을 위로 돌파하면 매수 신호, 아래로 내려오면 매도 신호입니다. 이동평균선보다 빠르게 전환점을 잡아냅니다. 볼린저밴드는 이동평균선 ±2σ 구간을 표시하는데, 밴드가 수축되면 큰 변동성 움직임이 임박했다는 신호입니다.

이 3가지가 us-stock 6팩터 중 기술적 분석 25%의 원재료입니다. RSI·MACD·MA 신호를 종합해서 기술 점수를 산출하고, 나머지 5개 팩터와 합산합니다. 단일 지표로 매매하면 오신호가 많지만, 여러 지표를 복합 점수로 만들면 신뢰도가 올라갑니다.

개별 종목 분석보다 시장이 그 종목을 어떻게 평가하는지가 더 중요합니다.
Slide 4-7
상대강도(RS)

상대강도(RS)는 종목 수익률에서 SPY 수익률을 뺀 값입니다. RS가 0보다 크면 그 종목이 시장 평균을 이기고 있다는 뜻, 즉 아웃퍼폼하고 있는 것입니다. 반대로 RS가 0 미만이면 시장보다 못하고 있는 것입니다. 기관 투자자들은 항상 RS 상위 종목에 자금을 집중합니다.

반 전체가 시험 볼 때 평균보다 높은 점수를 받는 학생처럼 생각하시면 됩니다. RS > 0이면 그 종목이 반 평균보다 성적이 좋은 것입니다. 기관은 언제나 반 평균보다 높은 성적표를 가진 학생(종목)만 사 모읍니다. 시장이 전체적으로 빠질 때도 상대적으로 덜 빠지는 종목이 RS 강한 종목입니다.

us-stock의 6팩터에서 RS는 15% 가중치를 갖습니다. RS > 0이고 이동평균 정배열이면 Trend 전략으로 분류됩니다. 가장 안정적인 추세 추종 방식입니다. 기관 매집 패턴과 RS가 동시에 좋은 종목이 시스템이 선호하는 타입입니다.

기관이 실제로 무엇을 사고 있는지는 13F 공시로 확인할 수 있습니다.
Slide 4-8
밸류에이션 핵심

주식 가격이 합리적인지 판단하는 핵심 도구는 PER입니다. 주가를 주당순이익(EPS)으로 나눈 값입니다. PER이 20이라면 지금 이익 기준으로 20년치를 미리 사는 것입니다. 중요한 구분은 Trailing PER(과거 이익 기준)과 Forward PER(미래 이익 전망 기준)입니다. 투자는 미래를 사는 것이니 Forward PER이 더 중요합니다.

현재 S&P 500 전체 Forward PER은 약 19.8배입니다. 역사적 평균이 16~17배이니 약간 높아 보이지만, AI 기업들의 이익 성장률을 감안하면 합리적인 범위입니다. EPS 상향 추세가 핵심입니다. Forward 12M EPS가 계속 올라가고 있다면, 주가가 올라도 PER이 낮아질 수 있습니다. ROE는 15% 이상이면 자본 효율이 우수한 기업입니다.

PER이 낮다고 무조건 좋은 것이 아닙니다. 이익이 빠지면 PER이 올라가기 때문입니다. "싸 보이는 주식"이 실제로는 이익이 급감하고 있는 종목일 수 있습니다. 반대로 PER이 높아도 이익 성장률이 그것을 정당화한다면 좋은 투자가 될 수 있습니다.

위험관리 — MDD와 Stop Loss

MDD(Maximum Drawdown) 비대칭성을 반드시 기억하십시오. 주식이 10% 떨어지면 원금 회복에 11.1% 상승이 필요합니다. 20% 손실은 25% 회복, 50% 손실은 100% 회복이 필요합니다. 손실이 커질수록 회복이 기하급수적으로 어려워집니다. 처음부터 낙폭을 제한하는 것이 위험관리의 핵심입니다.

Stop Loss는 3단계입니다. 첫 번째, 손실제한(-5% 자동 매도). 두 번째, 이익실현(+10% 일부 매도). 세 번째, Trailing Stop(최고가에서 일정 비율 하락 시 자동 매도). us-stock은 체제별로 이 기준이 자동으로 조정됩니다. risk_on: -10%, neutral: -8%, risk_off: -5%, crisis: -3%입니다.

Stop Loss는 안전벨트입니다. 불편하지만 사고가 났을 때 목숨을 구합니다. 위기 국면에서 -3%에 손절하는 것이 -30% 손해보다 훨씬 낫습니다. "첫 손절이 제일 손실이 작다" — 이 문장 하나만 오늘 가져가셔도 됩니다.

13F — 스마트 머니 추적

13F는 미국 SEC의 의무 공시입니다. 운용자산 1억 달러 이상의 기관 투자자는 분기 종료 후 45일 이내에 보유 종목과 수량을 전부 공개해야 합니다. 이 데이터를 보면 워런 버핏의 버크셔 해서웨이, 르네상스 테크놀로지 같은 세계 최고 기관이 무엇을 사고 팔았는지 알 수 있습니다.

기관 지분이 증가하는 종목은 Strong Accumulation, 즉 대형 자금이 매집하고 있는 상태입니다. 반대로 기관 지분이 감소하면 Distribution, 분산 매도 상태입니다. 기관이 사고 있는 종목과 개인이 사고 있는 종목을 비교해 보면 방향이 다를 때가 많습니다. us-stock의 13F 팩터(10%)가 이 데이터를 자동으로 수집하고 점수화합니다.

한 가지 중요한 원칙이 있습니다. Look-ahead bias 방지입니다. 13F는 분기가 끝나고 45일 후에 공개됩니다. us-stock은 공시일 기준으로만 데이터를 계산합니다. 공시 전의 데이터를 마치 알고 있는 것처럼 계산하면 백테스트 결과가 왜곡됩니다.

거래량 팩터가 특히 중요합니다. 기관이 대량으로 매집할 때 거래량이 갑자기 튑니다. 표준편차 기반으로 이상 거래량을 탐지해서 기관의 움직임을 포착합니다. 개인이 모르는 사이에 기관이 먼저 사고 있는 종목을 찾아내는 것입니다. 503개 종목에 이 계산을 매일 돌리면 Grade A~F로 등급화된 결과가 나옵니다.

Grade A는 80점 이상(Strong Accumulation), B는 62점 이상, C는 48점 이상, D는 35점 이상, F는 35점 미만(Capitulation)입니다. Verdict와 Grade를 교차하면 오늘의 Action이 결정됩니다. GO+A/B=BUY, CAUTION+A/B=SMALL BUY, STOP=HOLD 전체입니다.

점수 계산이 끝나면 상위 종목들을 AI가 심층 분석합니다.

Part 4 — AI 분석

AI 분석은 3개 모델을 순차적으로 씁니다. Gemini Flash(1순위, 가장 저렴) → GPT-4o-mini(2순위) → Perplexity Sonar(3순위, 실시간 웹검색). Fallback Chain이라고 부릅니다. 하나가 다운되어도 다음으로 자동 전환되어 파이프라인이 절대 중단되지 않습니다. AI가 종목마다 투자 보고서를 자동으로 써주는 것입니다.

생성 내용은 4가지입니다. 투자 thesis(1~2문단), Bull Catalysts 3개(상승 촉매), Bear Risks 2~3개(하락 리스크), 매수·매도 Recommendation. 하락 리스크를 반드시 포함하도록 강제합니다. AI가 장밋빛 분석만 내놓는 것을 방지하기 위해서입니다. 비용은 10개 종목 전체 분석에 하루 7~42원, 한 달 약 1,000~1,300원 수준입니다.

사람이 같은 수준의 분석을 하면 종목 하나에 30분~1시간이 걸립니다. 10개 종목이면 5~10시간입니다. AI가 초당 처리합니다. 사람의 역할은 AI 분석을 읽고 최종 판단을 내리는 것입니다. AI는 도구입니다.

Part 5 — ML 지수 예측

LightGBM 모델과 27개 피처로 SPY와 QQQ의 5일 방향(상승/하락)을 확률로 예측합니다. 6개 피처 카테고리가 있습니다. 가격·기술(RSI·MACD·MA), 추세·모멘텀, 거시·변동성(VIX·Fear&Greed), 신용·체제(HYG/IEF), 시장 폭(RSP/SPY), 앙상블(5/10/20일 가중)입니다. 이 27개가 모델에 입력되어 방향 확률이 계산됩니다.

결과는 DirectionCard 형태로 보여줍니다. 방향(BULLISH/BEARISH/NEUTRAL) + 신뢰도 % + Key Drivers Top 3입니다. 해석 기준은 간단합니다. 70% 이상이면 녹색 고신뢰, 50~70%는 노란색 중간, 50% 미만은 빨간색 신중입니다. SPY와 QQQ 둘 다 BULLISH 70% 이상이면 가장 강한 매수 신호입니다.

ML 모델은 walk-forward 검증을 거쳤습니다. 미래 데이터를 사용하는 look-ahead bias 없이 검증합니다. 단, ML 예측이 100% 맞을 수는 없습니다. 5센서 체제 + Grade + ML 예측, 세 가지를 모두 확인한 뒤 최종 결정을 내리시기 바랍니다.

Part 6 — 리스크 관리 7종 알림

7종 알림이 있습니다. VIX 급등 알림, Stop Loss 발동 알림, 드로다운 경고, 집중도 경고(특정 종목/섹터 쏠림), 체제 전환 알림, 섹터 쏠림 경고, 시스템 에러 알림입니다. 각 알림은 HIGH·MEDIUM·LOW 3단계로 심각도가 분류됩니다. HIGH는 즉각 조치, MEDIUM은 주시, LOW는 참고 수준입니다.

알림 3개 이상이 동시에 발생하면 포지션 축소를 검토합니다. /risk 페이지에서 활성 알림 전체를 한눈에 볼 수 있습니다. VaR(1일 최대 손실 추정), CDaR(극단적 손실 추정), Stress Test(2008 금융위기·2020 코로나 시나리오 시뮬레이션)도 이 페이지에서 확인합니다.

알림이 많다고 무조건 팔지 않아도 됩니다. 알림의 종류와 심각도를 보고 판단합니다. 예를 들어 섹터 쏠림 MEDIUM 알림 하나는 포지션 전체를 팔 이유가 되지 않습니다. 하지만 VIX 급등 HIGH + 체제 전환 HIGH가 동시에 오면 즉각 조치가 필요합니다.

Verdict × Grade 매트릭스

Verdict는 3종류입니다. GO는 Regime 정상 + Gate GO + ML Bullish 세 조건이 모두 충족될 때입니다. STOP은 Regime이 crisis 또는 risk_off이거나 Gate가 STOP일 때입니다. CAUTION은 그 외 모든 경우입니다. 이 Verdict와 종목별 Grade를 교차하면 오늘 할 일이 결정됩니다.

매트릭스입니다. GO + Grade A/B = BUY. GO + Grade C = WATCH. GO + Grade D/F = WATCH. CAUTION + Grade A/B = SMALL BUY. CAUTION + Grade C 이하 = WATCH. STOP + 전체 = HOLD. 이 매트릭스가 us-stock의 최종 결론입니다. 매일 아침 이 표에서 오늘 할 일이 결정됩니다.

중요한 원칙이 하나 있습니다. 시장이 위기 국면(STOP)이면 아무리 Grade가 좋은 종목도 사지 않습니다. 좋은 종목도 시장 전체가 하락할 때 함께 내려갑니다. 체제와 Grade 두 가지를 반드시 교차 확인하십시오.

/regime 페이지

/regime 페이지는 "시장이 지금 살 만한가"를 5가지 센서로 판단하는 화면입니다. 표시 내용은 4가지입니다. 체제 배지(risk_on·neutral·risk_off·crisis)와 종합 점수, 5센서 각각의 현재 상태(정상·주의·위험), Market Gate(GO·CAUTION·STOP), 섹터 히트맵(11개 섹터 색상)입니다.

판독 방법입니다. 빨간 센서가 3개 이상이면 시장 위험 신호입니다. Verdict가 GO여도 빨간 센서가 3개 이상이면 조심해야 합니다. Gate STOP이면 신규 매수는 금지입니다. 섹터 히트맵에서 초록이 많으면 시장이 전반적으로 건강한 상태입니다. 빨간 섹터가 많으면 특정 섹터에서 자금이 빠져나가고 있다는 신호입니다.

이것만 기억하십시오. 체제 배지가 crisis 또는 risk_off이거나, Gate가 STOP이면 그날은 신규 매수 금지입니다. 이 기준 하나만 지켜도 큰 손실을 막을 수 있습니다.

판독 방법입니다. Grade A + Action BUY = 가장 강한 매수 신호입니다. GO 체제에서만 나옵니다. Grade B + SMALL BUY = CAUTION 체제에서 소량 매수 권장입니다. Action이 WATCH이면 관찰 대상이고, HOLD이면 기존 보유는 유지하되 신규 매수는 하지 않습니다. Strategy를 보면 어떤 방식으로 접근할지 알 수 있습니다. Trend는 추세 추종, Swing은 단기 반등, Reversal은 역발상입니다.

핵심은 Verdict와 Grade 교차 확인입니다. Grade A가 나와도 Verdict가 STOP이면 HOLD입니다. 반드시 두 가지를 함께 보십시오. /regime 페이지에서 체제를 먼저 확인하고 이 페이지로 오는 것이 올바른 순서입니다.

이 페이지를 쓰면 "왜 이 종목이 Grade A인가"를 직접 확인할 수 있습니다. 모든 팩터가 균형 잡혀 있는 종목과 특정 팩터만 높은 종목은 투자 성격이 다릅니다. 팩터 분해를 보고 자신의 투자 스타일에 맞는 종목을 고르십시오.

알림이 많다고 무조건 팔지 않아도 됩니다. 알림의 종류와 심각도를 보고 판단하십시오. 집중도 경고 MEDIUM 하나는 큰 문제가 아닙니다. VIX 급등 HIGH + 체제 전환 HIGH 조합이 나오면 그때 즉각 조치하십시오.

/performance 페이지

/performance 페이지는 이 전략이 실제로 SPY를 이기고 있는지 숫자로 보여줍니다. 운동선수 기록표처럼 생각하시면 됩니다. 성적이 좋으면 전략을 신뢰하고, 나쁘면 파라미터를 재검토합니다. 핵심 지표는 4개입니다. Sharpe Ratio(1.0 이상이면 양호), Alpha(SPY 대비 초과 수익률), MDD(전략의 최대 낙폭), Win Rate(매수 신호 적중률)입니다.

판독 방법입니다. Sharpe 1.0 이상 + Alpha 양수 = 전략이 SPY를 이기는 중입니다. Win Rate 50% 이상 + Risk/Reward 2:1 이상 = 좋은 전략입니다. SPY 대비 수익률 커브가 우상향하면서 격차가 벌어지면 알파가 쌓이고 있는 것입니다. 반대로 전략이 SPY에 뒤처지면 어떤 팩터가 문제인지 /ml 페이지로 돌아가서 확인합니다.

백테스트 한계를 반드시 기억하십시오. 과거 수익률이 미래를 보장하지 않습니다. 백테스트 결과가 좋다는 것은 이 전략이 과거 데이터에서 작동했다는 뜻입니다. 시장 환경이 바뀌면 성과가 달라질 수 있습니다.

/graph 페이지

/graph 페이지는 종목 간 관계를 네트워크 그래프로 시각화합니다. 4개 탭이 있습니다. System View(전체 섹터-종목 네트워크), Stock View(특정 종목의 섹터·연관 종목), Market View(날짜별 Grade A/B 분포 변화), Ticker-Centric(선택 종목 중심 네트워크)입니다. 노드 크기는 점수, 색상은 Grade입니다. 초록=A, 파란색=B, 회색=C 이하입니다.

판독 방법입니다. 같은 섹터에 Grade A 노드가 밀집되어 있으면 그 섹터에서 테마 투자 흐름이 진행 중이라는 신호입니다. XLK(기술주) 섹터에 초록 노드가 많으면 IT 섹터 전반에 기관 자금이 들어오고 있다는 뜻입니다. Market View 탭에서 날짜별로 Grade A/B 종목 수의 변화를 보면 시장 건강도의 추이를 알 수 있습니다.

Ticker-Centric 탭이 가장 실용적입니다. 관심 종목을 선택하면 그 종목과 같은 섹터에 있는 종목들의 Grade를 한눈에 파악할 수 있습니다. NVDA를 중심으로 보면 반도체 섹터 전체의 Grade 분포가 보입니다.

이 시스템은 두 기기가 역할을 나눠 운영됩니다. Mac은 분석 전용입니다. Python 3.13 venv, 모든 API 키, 평일 07:00 launchd 자동 실행이 여기에 있습니다. 파이프라인 결과는 output/ 디렉토리에 저장되고, localhost:3000에서 개발용 대시보드를 확인합니다. Synology NAS는 서빙 전용입니다. Docker로 Next.js 대시보드를 실행하고 port 8889로 외부 접속을 받습니다. read-only 데이터 쿼리와 게시판(board.db) 쓰기만 담당합니다.

편의점 본사와 지점으로 비유하면 됩니다. 본사(Mac)에서 데이터 분석을 하고, 각 지점(NAS)은 화면만 보여줍니다. 코드가 바뀔 때는 GitHub Actions가 자동으로 처리합니다. git push → npm run build → Docker image 생성 → GHCR 업로드 → Synology에서 docker pull. 데이터 변경은 rsync만으로 처리합니다.

핵심 보안 원칙입니다. API 키는 Mac에만 있습니다. NAS에 절대 복사하지 않습니다. NAS가 해킹되더라도 Gemini·OpenAI·FRED API 키가 노출되지 않습니다. 이 원칙을 절대 바꾸지 마십시오.

Claude Code 기본 설정

Claude Code는 터미널에서 실행하는 AI 코딩 에이전트입니다. 핵심은 CLAUDE.md 파일입니다. 프로젝트 루트에 이 파일이 있으면 Claude Code가 매 대화 시작 시 자동으로 읽습니다. us-stock의 CLAUDE.md는 1,200줄 이상의 시스템 설명서입니다. 아키텍처 구조, 금지 패턴, 파이프라인 실행 순서, 환경 변수, 알려진 버그 목록이 전부 들어있습니다. Claude Code는 이것을 읽고 프로젝트를 이해한 뒤 작업합니다.

~/.claude/settings.json은 전역 설정 파일입니다. autoApproveTools(도구 자동 승인), model(사용 모델), permissions(파일 접근 권한)을 설정합니다. /settings 명령어로 UI를 열어 직접 바꿀 수 있습니다. ~/.claude/ 디렉토리 구조는 CLAUDE.md(전역 지침), settings.json(설정), output-styles/(응답 스타일), skills/(재사용 워크플로우)로 구성됩니다.

전역 CLAUDE.md(~/.claude/CLAUDE.md)는 모든 프로젝트에 적용되고, 프로젝트 CLAUDE.md(./CLAUDE.md)는 그 프로젝트에만 적용됩니다. us-stock의 CLAUDE.md를 열어보시면 전체 시스템 설명서가 담겨있습니다. 이 파일을 잘 관리하는 것이 Claude Code를 효과적으로 쓰는 핵심입니다.

Claude Code의 응답 방식을 상황에 맞게 바꿀 수 있습니다. 세 가지 기본 스타일이 있습니다. Default는 작업 완수 효율성 중심입니다. 코드만 수정하고 설명을 최소화합니다. 빠른 작업에 적합합니다. Explanatory는 구현 근거와 코드 패턴을 심층 분석합니다. 아키텍처 리뷰나 팀 공유 시 유용합니다. Learning은 AI가 답을 주는 대신 단계별 가이드를 제공합니다. TODO(human) 마커로 직접 완성하도록 유도합니다. 핸즈온 워크숍에 적합합니다.

커스텀 스타일도 만들 수 있습니다. ~/.claude/output-styles/senior-review-mode.md 파일을 만들면 /senior-review-mode로 호출할 수 있습니다. 코드 리뷰 전문가 모드, 보안 감사 모드 등 원하는 스타일을 텍스트 파일로 정의하면 됩니다.

주의사항이 있습니다. 커스텀 스타일 파일은 기본 시스템 지침 전체를 교체합니다. 파일 편집 권한을 명시하지 않으면 Claude가 코드 수정 기능을 잃습니다. 커스텀 스타일 파일에 반드시 "파일 읽기·쓰기·수정 권한 있음"을 명시하십시오.

Slide 6-3
Skill — 모듈형 지능 주입

Skill은 재사용 가능한 워크플로우를 SKILL.md 파일로 저장하는 것입니다. 레시피북처럼 생각하시면 됩니다. Claude Code에게 파스타 만들기 레시피를 한 번 알려주면, 이후에는 "파스타 만들어줘"만 해도 매번 동일한 품질로 실행합니다. 매번 긴 설명을 반복할 필요가 없습니다.

SKILL.md 구조는 두 부분입니다. YAML frontmatter(name·description·invoke·allowed-tools)와 지침 내용입니다. 수동 호출은 /skill-name 형식입니다. 자동 호출은 Claude가 대화 맥락을 분석해서 스스로 로드합니다. $ARGUMENTS로 입력값을 전달합니다. 저장 위치는 .claude/skills/(프로젝트 전용)과 ~/.claude/skills/(전역 공용) 두 곳입니다.

실용적 장점은 세 가지입니다. 컨텍스트 오버헤드 감소(매번 긴 설명 불필요), 조직 지식 캡처(팀 공통 워크플로우를 파일로 보관), allowed-tools로 도구 권한 제어(Skill마다 허용 도구를 다르게 설정 가능). us-stock에도 여러 Skill이 정의되어 있습니다. /run-analysis, /check-regime 같은 것들입니다.

여러 Claude를 팀으로 운영하는 에이전트 아키텍처를 알아봅니다.
Slide 6-4
서브에이전트 — 독립 컨텍스트

Claude Code는 단일 에이전트만이 아닙니다. 서브에이전트를 생성해서 독립적으로 작업을 처리할 수 있습니다. 메인 에이전트는 사용자와 대화하고 전체 전략을 수립합니다. 지휘관 역할입니다. 서브에이전트는 독립적인 컨텍스트에서 임무를 완수하고 소멸합니다. 전문 직원 역할입니다.

내장 서브에이전트는 3종류입니다. Explore(Haiku 모델, 빠른 탐색·검색), Plan(설계·계획 수립), General-purpose(범용 작업). 메모리 스코프는 3계층입니다. User-level(~/.claude/CLAUDE.md, 모든 프로젝트에 적용), Project-level(./CLAUDE.md, 해당 프로젝트만), Local-level(.claude/settings.local.json, git 미추적, 개인 설정).

서브에이전트의 핵심 장점은 컨텍스트 격리입니다. 각 서브에이전트가 독립 컨텍스트를 가지므로 정보가 혼선을 일으키지 않습니다. 병렬 실행으로 속도를 극대화할 수 있습니다. 도구 권한도 서브에이전트마다 다르게 설정할 수 있습니다. 팀장은 전략만, 직원은 실행만 담당하는 구조입니다.

서브에이전트들이 팀으로 협업하는 에이전트 팀을 알아봅니다.
Slide 6-5
에이전트 팀 — 수평적 협업

한 명의 Claude가 혼자 하는 것이 아니라 여러 Claude가 역할을 분담해서 동시에 작업하는 것이 에이전트 팀입니다. 팀 구성 패턴은 이렇습니다. 기획자(Planner) — 요구사항을 설계로 변환. 구현자(Coder) — 설계를 코드와 테스트로 변환. 악마의 대변인(Critic) — 코드 품질·보안·엣지 케이스 검토. 이 3명이 순차·병렬로 작업합니다.

활성화 방법은 환경변수 하나입니다. export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1. in-process(같은 터미널에서 병렬 실행)와 split panes(여러 터미널)을 선택할 수 있습니다. us-stock의 21개 이상 에이전트 생태계가 바로 이 원리입니다. equity-lead가 지휘하고, equity-factor-builder가 구현하고, gbm-code-reviewer가 검토하는 구조입니다.

팀장 혼자 하는 것과 3명이 분업하는 것의 차이는 큽니다. 3명이 동시에 작업하면 시간이 단축되고, 각자 전문 분야에 집중하므로 품질이 올라갑니다. Claude Code 에이전트 팀을 쓰면 개인 개발자도 팀 수준의 작업 품질을 낼 수 있습니다.

💬 청중 참여
"팀장 혼자 전략·개발·리뷰를 다 하는 것과 3명이 분업하는 것의 차이를 경험해 보신 분 계십니까? Claude Code 팀도 같은 원리입니다."

Slide 6-6
Plugin — 재사용 패키지

Plugin은 Skills·Commands·Agents·Hooks를 하나의 패키지로 묶어 배포하는 것입니다. 스마트폰 앱처럼 생각하시면 됩니다. 남이 만든 앱을 1줄로 설치하고, 내가 만든 앱을 배포해서 팀이 공유합니다. 설치는 claude plugin install <url> 한 줄입니다.

Plugin 구조입니다. .claude-plugin/ 디렉토리 안에 plugin.json(메타데이터), skills/(재사용 워크플로우), commands/(슬래시 명령어), agents/(전문 에이전트), hooks/(이벤트 반응)가 들어있습니다. 설치 후에는 /plugin:skill-name 형식으로 호출합니다. 네임스페이스가 분리되어 충돌이 없습니다. GitHub 리포로 배포하면 팀 전체가 사용할 수 있습니다.

4종류의 Plugin 컴포넌트입니다. Skills(재사용 워크플로우), Commands(슬래시 명령어 추가), Agents(전문 에이전트 정의), Hooks(PreToolUse·PostToolUse 등 이벤트에 반응). 이 4가지를 조합하면 원하는 기능을 완전히 캡슐화할 수 있습니다.

Plugin 중에 특히 유용한 3가지 명령어를 소개합니다.
Slide 6-7
/simplify — 코드 품질 마무리

/simplify는 구현 완료 후 코드를 정리하는 도구입니다. 3개 에이전트가 병렬로 작업합니다. 에이전트 1(Reuse) — 중복 코드를 공통 함수로 추출. 에이전트 2(Quality) — SOLID 원칙 위반을 탐지하고 개선안 제안. 에이전트 3(Efficiency) — 불필요한 루프·변수 제거. 3개가 동시에 작업하고 결과를 통합해서 최종 개선안을 제시합니다.

사용 방법입니다. /simplify (기본 전체 정리), /simplify focus on memory efficiency (메모리 효율 중심), /simplify focus on database queries (DB 쿼리 최적화). PR 제출 전 최종 점검 용도로 씁니다. 팀 코딩 컨벤션을 자동으로 적용할 수도 있습니다.

편집장 비유가 딱 맞습니다. 기자(개발자)가 초고를 쓰면, 편집장(/simplify)이 군더더기를 빼고 논리를 다듬어 최종본을 만드는 것입니다. 작동하는 코드를 깔끔한 코드로 바꾸는 마지막 단계입니다.

대규모 변경이 필요할 때는 /batch를 씁니다.
Slide 6-8
/batch — 대규모 리팩토링

/batch는 대규모 변경을 안전하게 분해해서 실행합니다. 5단계 흐름입니다. 1단계 조사(Explore) — 전체 코드베이스를 파악합니다. 2단계 계획(Plan) — 작업을 N개의 독립 단위로 분해합니다. 3단계 승인 — 사용자가 계획을 검토하고 확정합니다. 4단계 격리 실행 — Git Worktree로 각 작업을 별도 브랜치·별도 디렉토리에서 실행합니다. 5단계 PR 배달 — 작업별로 PR을 생성합니다.

핵심 장점은 격리입니다. 하나의 작업이 실패해도 나머지에 영향을 주지 않습니다. 5~30개 작업을 동시에 실행할 수 있어서 속도가 극적으로 빨라집니다. 작업별 PR이 생성되므로 리뷰가 쉽습니다. 전체 의존성 업그레이드나 수십 개 파일 일괄 리팩토링에 적합합니다.

건설 현장 관리자 비유입니다. 전체 공사를 독립 구역으로 나눠서 팀별로 동시에 짓고, 완성된 구역부터 검수(PR)하는 방식입니다. "한 번에 하나씩" 대신 "여러 개를 동시에, 안전하게"가 /batch의 원리입니다.

가장 강력한 도구인 Ralph를 알아봅니다.
Slide 6-9
Ralph — 멈추지 않는 루프

Ralph는 자율 실행 루프입니다. 완료 조건이 충족될 때까지 스스로 반복 실행합니다. 핵심 메커니즘은 stop-hook.sh 파일입니다. 매 실행 후 완료 조건을 확인하고, COMPLETE 문자열이 감지되면 루프를 종료합니다. --completion-promise로 완료 시 출력 문자열을 지정하고, --max-iterations 50으로 최대 반복 횟수를 제한해서 무한 루프를 방지합니다.

Best for(적합한 경우): 단위 테스트가 전부 통과할 때까지 자율 수정, 버그 수정 사이클 반복, 대량 데이터 처리. Worst for(부적합한 경우): 외부 API 호출이 많은 작업(비용 예측 불가), 되돌리기 어려운 변경(DB 삭제 등). 자동화된 품질 검사관처럼 작동합니다. 공장 라인 끝에서 제품이 기준에 맞을 때까지 계속 재작업을 지시합니다.

실용적 사례입니다. "테스트 100개가 전부 통과할 때까지 알아서 고쳐줘" — Ralph가 실행-실패-수정-재실행을 반복하다가 전부 통과되면 자동 종료됩니다. 개발자가 자리를 비워도 됩니다. 돌아왔을 때 결과가 나와 있습니다.

에이전트 갤러리 — /ai-builder

대시보드 /ai-builder 페이지에서 에이전트를 웹 브라우저로 실행합니다. 6개 주요 에이전트입니다. service-evolver(매일 1개 자동 개선, 자동 실행), perf-lead(Sharpe/Alpha 개선 총괄, 전략 수익률이 SPY를 하회할 때 실행), equity-lead(80개 이상 팩터 상관관계 감사, 팩터 추가·수정 후 실행), macro-lead(거시 피처 총괄, 금리·VIX 급변 시), research-lead(arXiv 논문 기반 신규 팩터 발굴, 성능 정체 시), ai-lead(AI 분석 품질·API 비용 관리, 주간 점검).

기본 실행 3단계

Claude Code 관점에서 실행 순서를 정리합니다. Step 1: source .venv/bin/activate — Python 환경을 켭니다. Step 2: python scripts/run_integrated_analysis.py — Phase 0~3 전체 파이프라인 자동 실행, 5~15분 소요. Step 3: cd frontend && npm run dev — 대시보드 서버 시작, localhost:3000 접속.

AI 분석 비용 구조입니다. Gemini Flash(1순위): 입력 $0.10/1M tokens, 출력 $0.40/1M tokens. 가장 저렴합니다. GPT-4o-mini(2순위): 입력 $0.15, 출력 $0.60. Gemini 실패 시 자동 전환됩니다. Perplexity Sonar(3순위): $3/1K requests. 실시간 웹검색이 가능하지만 가장 비쌉니다. 시스템 일일 예산 상한은 $0.20입니다. APIUsageTracker가 자동으로 추적하고 예산 초과 시 알림을 보냅니다.

실제 비용입니다. 10개 종목 전체 AI 분석 시 하루 7~42원, 한 달 약 1,000~1,300원입니다. 비용 절감 팁은 하나입니다. Top 20 종목만 AI 분석하면 비용이 90% 절감됩니다. 503개 전체를 AI 분석할 필요는 없습니다. Grade A/B 상위 20개만 심층 분석해도 충분합니다. /costs 페이지에서 일별·월별 비용을 추적할 수 있습니다.
