Slide 1-6
용어 정리 — FO · MO · FOMO · SaaS · PaaS

강의에서 자주 나오는 약어들 정리합니다.

FO (Fear Of) /fiːr əv/ [피어 오브] — ~에 대한 두려움. 주식에서 "FO 매도"는 두려움에 파는 것.

MO (Modus Operandi) /ˈmoʊdəs ˌɒpəˈrændaɪ/ [모더스 오퍼랜다이] — 특유의 행동 패턴. "그게 그 사람 MO야" = 그 사람 특유의 방식.

FOMO (Fear Of Missing Out) /ˈfoʊmoʊ/ [포모] — 나만 뒤처질까봐 두려운 심리. 급등주 추격 매수가 대표적.

SaaS (Software as a Service) /sæs/ [새스] — 소프트웨어를 구독으로 제공. 넷플릭스, 슬랙, 노션이 다 SaaS.

PaaS (Platform as a Service) /pæs/ [패스] — 개발 플랫폼을 클라우드로 제공. AWS 같은 것들.

SaaS 칼립소 — 클로드 코드 등장 이후 "내가 직접 만들 수 있다"는 인식이 퍼지면서 SaaS 기업 주가가 흔들리는 현상. 2025년 2월 나스닥 폭락의 한 원인.

이제 2025년 2월, 나스닥이 왜 흔들렸는지 그 배경을 보겠습니다.

클로드 코드 출시와 2025년 2월 나스닥

2025년 2월. 클로드 코드가 일반 공개됐습니다. 그 직후 나스닥이 흔들렸어요.

왜일까요? 시장이 계산을 한 겁니다. "누구나 소프트웨어를 직접 만들 수 있는 시대가 오면, 월 $50~$500씩 구독료 받던 SaaS 회사들은 어떻게 되나?"

Notion, Figma, Slack, Asana — 이 회사들이 파는 게 뭡니까? 결국 소프트웨어예요. 그 소프트웨어를 이제 개인이 AI로 직접 만들 수 있다면, 구독할 이유가 줄어드는 거죠.

이게 SaaS 칼립소입니다. 단순한 주가 조정이 아니에요. 비즈니스 모델 자체에 대한 시장의 의문입니다.

그리고 이 흐름은 반대로 기회이기도 합니다. AI 인프라를 파는 회사들 — 엔비디아, AMD, TSMC, AWS, Azure — 이들은 오히려 수혜를 받습니다. 더 많은 사람이 AI를 쓸수록 더 많은 인프라가 필요하니까요.

이게 오늘 S04에서 다룰 AI 투자 흐름의 핵심입니다.

Claude Code는 단일 에이전트만이 아닙니다. 여러 Claude가 역할을 분담해서 동시에 작업하는 에이전트 팀을 만들 수 있습니다.

팀 구성 패턴은 이렇습니다. 기획자(Planner) — 요구사항을 설계로 변환. 구현자(Coder) — 설계를 코드로 변환. 검토자(Critic) — 코드 품질·보안·엣지 케이스 검토. 이 3명이 순차·병렬로 작업합니다.

에이전트 팀은 수직 지휘 구조가 아닙니다. 수평 협업 구조입니다. 각 에이전트가 전문 분야를 맡고, 결과를 서로에게 넘겨줍니다.

Slide IT-1
왜 맥인가 — CLI vs GUI

클로드 코드는 Bash Shell 기반으로 동작합니다. 이게 왜 중요하냐면, 클로드 코드가 컴퓨터 안에서 뭔가를 실제로 하려면 명령어를 직접 실행해야 하거든요.

GUI, 즉 마우스로 클릭하는 방식은 자동화가 어렵습니다. 우리가 마우스로 클릭해서 폴더 만들고 파일 이동하는 건 사람이 직접 해야 해요. 근데 CLI, 즉 커맨드라인 방식은 이걸 명령어 한 줄로 처리합니다. 그리고 여러 명령어를 순서대로 파일에 적어두면 — 그게 Shell Script입니다. 자동화의 원리예요.

맥은 Unix 기반입니다. 리눅스랑 유사한 환경이라 이 Bash Shell이 내장되어 있어요. 터미널 열면 바로 됩니다. 윈도우는 파워쉘이 있긴 한데, 클로드 코드 쓰기에 불안정한 부분이 있습니다. 그래서 신기능이 맥에서 먼저 나오고, 안정화 되면 윈도우로 넘어가는 패턴이 반복됩니다.

결론: 클로드 코드를 제대로 쓰려면 맥이 필요합니다. 올해 안에 맥북, 클로드 코드, 옵시디언 — 이 세 가지는 하셔야 합니다.

💬 청중 참여
"지금 맥북 있으신 분 손 들어보세요. — 윈도우 쓰시는 분은 오늘 이후에 진지하게 맥 생각해 보시기 바랍니다. 이유는 오늘 다 설명드리겠습니다."

Slide IT-2
Workflow 자동화의 설계도

Workflow는 말 그대로 "일이 흘러가는 순서"입니다. 예전엔 사람이 이 순서를 직접 따랐어요. 주가 확인 → 뉴스 검색 → 차트 분석 → 엑셀 정리. 이게 매일 반복이었죠.

Shell Script가 처음 나온 게 60년 전입니다. 그때부터 사람들은 "이 반복 작업을 자동화하고 싶다"고 생각했어요. 명령어들을 순서대로 파일에 적어두면 버튼 하나로 다 실행되는 겁니다.

지금 AI 시대에는 이 Workflow를 AI가 스스로 판단하며 실행합니다. 제 시스템이 매일 새벽에 자동으로 500종목 분석하고 보고서까지 만들어놓는 게 바로 이 원리예요.

Workflow = 순서 정의. Agent = 그 순서를 AI가 스스로 판단하며 실행하는 것.

Slide IT-3
IT 개념 5분 정리 — 왜 알아야 하는가

여러분, 잠깐 저한테 5분만 주십시오. 이 5분 동안 IT 개발자들이 매일 쓰는 핵심 개념들을 설명해 드리겠습니다. 코딩을 모르셔도 됩니다. 개념만 알면 충분합니다.

왜 IT 개념을 주식 투자 강의에서 다루느냐고 하시겠죠. 이유는 간단합니다. 2026년 현재 가장 돈이 몰리는 곳이 AI 인프라입니다. 엔비디아, AMD, TSMC, AWS, Azure — 이 기업들이 무슨 사업을 하는지 구조적으로 이해하면 투자 판단이 달라집니다.

지금부터 설명드릴 내용은 OS, CPU, 웹, 클라우드입니다. 어렵지 않습니다. 오히려 이걸 알고 나면 '아, 그래서 엔비디아가 그렇게 비싸구나'가 이해됩니다.

가장 기본부터 시작합니다. 컴퓨터를 켜면 제일 먼저 실행되는 것, OS입니다.
Slide IT-4
OS 커널 구조 — 하드웨어의 문지기

컴퓨터를 켜면 가장 먼저 실행되는 것이 운영체제, 즉 OS입니다. OS는 두 부분으로 나뉩니다. 커널과 사용자 공간입니다.

커널은 운영체제의 핵심입니다. CPU, 메모리, 디스크에 직접 접근합니다. 우리가 쓰는 앱들은 커널 위에서 실행됩니다. 앱이 파일을 읽거나 네트워크에 연결할 때는 커널에게 요청합니다. 이 요청을 시스템 콜이라고 합니다.

비유하자면 커널은 건물 관리자입니다. 우리는 각 층에 살고 있고, 전기를 쓰거나 물을 쓰려면 관리자에게 요청해야 합니다. 직접 전기실에 들어갈 수 없는 것처럼, 앱이 직접 하드웨어를 만질 수 없습니다. 커널이 그 역할을 대신합니다.

그러면 그 하드웨어 중 핵심인 CPU와 메모리를 보겠습니다.
Slide IT-5
CPU + 메모리 구조 — AI 시대에 GPU가 왜 비싼가

CPU는 연산을 담당합니다. 제어장치는 '무엇을 할지' 결정하고, 연산장치는 실제 계산을 합니다. CPU 안에는 레지스터라는 초고속 임시 저장소가 있습니다.

메모리는 계층 구조입니다. CPU 레지스터가 가장 빠릅니다. 그 다음 캐시 메모리, RAM, SSD/HDD 순입니다. 빠를수록 용량이 작고 비쌉니다. 느릴수록 대용량이고 저렴합니다.

이게 왜 투자에 중요합니까? AI 학습은 엄청난 양의 행렬 연산을 합니다. CPU는 직렬 처리, GPU는 병렬 처리에 특화되어 있습니다. GPU 코어 수가 CPU보다 수십 배 많습니다. 이 차이가 AI 시대에 엔비디아가 독주하는 이유입니다. H100 하나가 3만 달러인 이유입니다.

컴퓨터가 여러 프로그램을 동시에 실행하는 원리를 볼 차례입니다.
Slide IT-6 ⭐
Context Switching — 멀티태스킹의 비밀

컴퓨터는 여러 프로그램을 '동시에' 실행합니다. 지금 이 강의를 들으시면서 음악도 틀고 메모도 하시는 분 계십니까? 이게 어떻게 가능할까요?

비밀은 Context Switching입니다. CPU는 사실 한 번에 하나만 처리합니다. 그런데 너무 빠르게 전환하기 때문에 동시에 하는 것처럼 보입니다. 1초에 수천 번씩 '지금은 크롬 처리, 지금은 카카오톡 처리, 지금은 음악 앱 처리'를 반복합니다.

이때 PCB가 중요합니다. PCB는 프로세스의 현재 상태를 저장해 두는 서류철입니다. '어디까지 했는지, 레지스터 값이 뭔지' 저장해 두고, 나중에 돌아올 때 그걸 보고 이어서 실행합니다.

💬 청중 참여
"지금 이 순간 여러분 스마트폰에 몇 개의 앱이 실행 중입니까? 한번 확인해 보십시오. 10개? 20개? — 이 모든 것을 CPU 하나가 처리합니다. Context Switching 덕분입니다."
프로세스 상태 머신으로 이해해 보겠습니다.
Slide IT-7
프로세스 상태 머신 — NEW에서 TERMINATED까지

프로세스가 거치는 상태는 총 다섯 가지입니다. NEW → READY → RUNNING → WAITING → TERMINATED.

프로그램을 실행하면 NEW 상태로 생성됩니다. CPU 스케줄러가 선택하면 RUNNING 상태. 실행 중에 파일을 읽거나 네트워크 응답을 기다려야 한다면 WAITING 상태로 전환됩니다. I/O 작업이 완료되면 다시 READY로 돌아옵니다.

왜 WAITING 상태가 따로 필요할까요? CPU는 나노초 단위로 작동하지만 I/O 디스크는 밀리초 단위입니다. 10만 배 차이입니다. 기다리는 동안 CPU를 붙잡고 있으면 낭비입니다. WAITING으로 빼고 다른 프로세스에게 CPU를 넘겨주는 것이 훨씬 효율적입니다.

Thread vs Process, 메모리 공유가 핵심 차이입니다.
Slide IT-8
Thread vs Process — 메모리 공유가 핵심 차이

Process는 독립적인 실행 단위입니다. 각자 자신만의 메모리 공간을 가집니다. Thread는 Process 안에서 실행되는 더 작은 단위입니다. 차이는 메모리 공유 여부입니다.

같은 Process 안의 Thread들은 CODE, DATA, HEAP 메모리를 함께 씁니다. 단, STACK은 Thread마다 개별적으로 가집니다. 이 공유 덕분에 Thread 간 통신이 빠릅니다.

실제 예를 들면, Chrome은 탭마다 독립 Process를 씁니다. 탭 하나가 멈춰도 다른 탭은 살아있습니다. us-stock 파이프라인도 Phase별로 독립 Process로 실행됩니다. 하나가 실패해도 다른 Phase는 영향받지 않는 이유입니다.

웹 통신의 기초, URL과 HTTP입니다.
Slide IT-9
WEB 기초 — URL · HTTP · 상태코드

웹의 기초를 설명드리겠습니다. URL은 '어디에 있는 자원인지'를 표시합니다. https://example.com/path 에서 https는 프로토콜, example.com은 서버 주소, /path는 자원의 위치입니다.

HTTP는 요청과 응답의 규약입니다. 요청할 때 Method가 있습니다. GET은 데이터 조회, POST는 데이터 전송, PUT은 수정, DELETE는 삭제입니다. 응답에는 상태 코드가 있습니다. 200은 성공, 404는 '없음', 500은 서버 오류입니다.

여러분이 유튜브에서 영상을 클릭하는 것, 쇼핑몰에서 주문하는 것, 뱅킹 앱을 쓰는 것 — 모두 이 HTTP 요청/응답으로 작동합니다. 눈에 보이지 않지만 초당 수백만 번씩 일어나고 있습니다.

웹 서비스 3-Tier 아키텍처를 보겠습니다.
Slide IT-10
웹서비스 3-Tier 아키텍처 — 프론트·백엔드·DB

웹 서비스는 크게 세 층으로 나뉩니다. 첫 번째 층은 프론트엔드입니다. 여러분의 브라우저에서 실행됩니다. 화면을 그리고 버튼 클릭을 처리합니다. HTML, CSS, JavaScript로 만들어집니다.

두 번째 층은 백엔드입니다. 서버에서 실행됩니다. 프론트엔드의 요청을 받아서 비즈니스 로직을 처리합니다. '이 사용자가 로그인됐는지', '이 주문이 유효한지'를 판단합니다.

세 번째 층은 데이터베이스입니다. 데이터를 영구적으로 저장합니다. us-stock 시스템도 이 구조입니다. Next.js 대시보드가 프론트엔드, Python 분석 서버가 백엔드, SQLite가 데이터베이스입니다.

Slide IT-REF-1
네트워크 핵심 — L2·L3·L4 · TCP vs UDP (참고)

네트워크 계층별 식별자를 정리합니다. L2 계층에서는 MAC 주소로 NIC를 식별합니다. L3 계층에서는 IP 주소로 호스트를 식별합니다. 라우터가 목적지를 찾을 때 씁니다. L4 계층에서는 포트 번호로 서비스를 식별합니다. HTTP는 80번, HTTPS는 443번, SSH는 22번입니다.

TCP는 연결 지향 프로토콜입니다. 3-way handshake로 연결을 맺고 순서를 보장합니다. HTTP, 파일 전송에 씁니다. UDP는 비연결 프로토콜입니다. 보장 없이 빠르게 전송합니다. DNS 조회, 스트리밍, 게임에 씁니다. 신뢰성이 필요하면 TCP, 속도가 우선이면 UDP입니다.

Slide IT-REF-2
OS 심화 — 메모리 레이아웃 · Compile 과정 (참고)

프로세스의 메모리는 네 영역으로 나뉩니다. CODE 영역은 실행 코드를 저장합니다. 읽기 전용입니다. DATA 영역은 전역 변수와 static 변수를 저장합니다. HEAP은 동적 할당 메모리이며 위 방향으로 자랍니다. STACK은 함수 호출과 지역 변수이며 아래 방향으로 자랍니다.

Compile 과정은 5단계입니다. .c 소스파일 → Preprocessor → Compiler(어셈블리) → Assembler(오브젝트 파일) → Linker(최종 실행 파일). Python이나 JavaScript는 인터프리터 방식으로 이 과정 없이 바로 실행합니다.

API vs MCP (참고)

API는 Application Programming Interface의 약자입니다. 두 프로그램이 서로 대화하는 방법입니다. REST API는 웹에서 가장 널리 쓰이는 API 설계 방식입니다. URI로 자원을 표현하고, HTTP Method로 행위를 표현합니다.

MCP는 Model Context Protocol입니다. Anthropic이 만든 새로운 표준으로, AI가 외부 도구와 대화하는 방법입니다. API가 프로그램끼리의 약속이라면, MCP는 AI가 도구를 쓰는 방법의 약속입니다.

us-stock 시스템에서 API가 세 곳에 씁니다. 프론트엔드가 백엔드에게 데이터를 요청할 때, Yahoo Finance에서 주가를 받아올 때, Gemini에게 AI 분석을 요청할 때입니다. Claude Code가 외부 도구(Telegram, Obsidian 등)를 쓸 때는 MCP를 통합니다.

Slide IT-REF-4
데이터베이스 — data.db가 뭔가 (참고)

데이터베이스는 데이터를 영구적으로 저장하고 빠르게 검색할 수 있는 시스템입니다. us-stock은 SQLite를 씁니다. 파일 하나(data.db)가 전체 데이터베이스입니다. 서버 없이 파일로 동작하는 경량 DB입니다.

저장되는 데이터는 크게 세 종류입니다. 주가 데이터(503개 종목 OHLCV 1년치), 분석 결과(Grade, Score, Action), 시스템 로그(파이프라인 실행 이력). 대시보드는 이 data.db를 읽어서 화면을 그립니다.

SQLite를 쓰는 이유는 간단합니다. 개인 프로젝트에 PostgreSQL 같은 무거운 DB는 과합니다. 파일 하나로 관리가 쉽고, 백업도 파일 복사로 끝납니다. NAS에서 Docker로 실행할 때도 볼륨 하나만 마운트하면 됩니다.

AI 투자 데이터 흐름도 (참고)

AI 투자 데이터의 흐름을 정리합니다.

[Wikipedia/Yahoo/FRED] → 데이터 수집(collectors/) → data/*.csv → 시장 체제 분석(market_regime.py) → regime_config.json → 스마트 머니 스크리닝(smart_money_screener_v2.py) → picks/*.csv → AI 분석(ai_summary_generator.py) → ai_summaries.json → 최종 리포트(final_report_generator.py) → final_top10_report.json → 대시보드(frontend/) → 사용자.

이 흐름의 각 단계가 하나의 Python 파일입니다. 각 파일이 독립적으로 실행되고, 결과를 JSON이나 CSV로 저장합니다. 다음 단계는 그 파일을 읽어서 처리합니다. 단순하지만 강력한 구조입니다.

API & REST API

주식 섹션에 들어오기 전에, 잠깐 기술 용어 두 가지만 짚고 가겠습니다. 이걸 모르면 왜 엔비디아가 비싸고, 왜 AWS가 돈을 버는지 이해하기 어렵습니다.

마지막으로 표에 있는 JVM, Servlet, Container, Framework는 자바 기반 백엔드 기술 스택입니다. 금융권 시스템 대부분이 자바로 돌아갑니다. 이 단어들이 뉴스에 나오면 이제 "아, 서버 이야기구나" 하고 느끼시면 됩니다.

Slide 5-2
Docker & Container

Docker와 Container입니다. 이게 왜 투자에 관계있냐? AWS, Azure, Google Cloud가 돈을 버는 핵심 단위가 바로 Container이기 때문입니다.

Image는 설계도입니다. 밀키트예요. 어떤 소프트웨어를 실행하려면 무엇이 필요한지 다 담겨있습니다. Container는 그 설계도를 바탕으로 실제 실행된 인스턴스입니다. 밀키트로 만든 실제 요리죠. 하나의 Image로 Container를 수십 개, 수백 개 동시에 실행할 수 있습니다.

VM(가상머신)과 비교하면, VM은 컴퓨터 전체를 가상화합니다. OS까지 통째로. 그래서 이미지가 수 GB고 시작에 수 분이 걸립니다. Container는 프로세스만 격리합니다. 수십~수백 MB에 수 초면 뜹니다.

ChatGPT 쓸 때 전 세계에서 동시에 수억 명이 씁니다. 그 요청들이 각각 Container에서 처리됩니다. 요청 많으면 Container 늘리고, 줄면 줄입니다. 이게 클라우드 과금 단위예요. Container가 많이 돌수록 AWS, Azure 매출이 올라갑니다.

💬 청중 참여
"넷플릭스 스트리밍 서비스, 전 세계 2억 명이 동시에 씁니다. 그 트래픽 하나하나가 Container로 처리됩니다. AWS에서 돌아가고요. 그래서 넷플릭스가 잘 될수록 AWS 매출도 올라가는 겁니다."
그럼 이 Container들이 돌아가는 클라우드 인프라가 AI 때문에 어떻게 바뀌고 있는지 보겠습니다.
Slide 5-3
클라우드와 AI 인프라

AI는 두 단계로 나뉩니다. 훈련과 추론입니다.

훈련은 모델을 만드는 과정입니다. GPT-4 하나 만드는 데 H100 GPU 수천 개를 수주일 동안 쉬지 않고 돌립니다. 비용이 1,000억 원을 넘어요. 이걸 할 수 있는 조직이 Meta, Google, OpenAI, Microsoft, Anthropic 정도입니다. 훈련용 GPU 수요는 이 회사들이 다 삽니다. 엔비디아 매출의 핵심이 여기서 나옵니다.

추론은 학습된 모델로 실시간 답변을 만드는 과정입니다. 여러분이 ChatGPT에 물어볼 때마다 추론이 일어납니다. 하루 수억 건. 이건 훈련과 달리 영구적으로 계속 일어납니다. 장기적으로 보면 추론 시장이 훈련보다 훨씬 큽니다.

투자 관점에서 핵심은 이겁니다. AI 훈련과 추론을 위해서는 GPU, 메모리, 서버, 냉각, 전력이 모두 필요합니다. 이 연쇄가 투자 기회의 지도입니다. 지금 시장에서 가장 확실한 테마입니다.

💬 청중 참여

시작은 AI 수요 폭발입니다. LLM 훈련, 실시간 추론, 멀티모달 확산. 이게 트리거입니다.

이 수요가 네 가지를 동시에 끌어올립니다. 첫째, 데이터센터. AI GPU를 수용할 공간, 전력, 냉각이 필요합니다. EQIX, DLR 같은 데이터센터 REIT가 수혜를 받습니다. 둘째, 반도체. NVDA, AMD가 GPU를 공급합니다. 셋째, 메모리. GPU 한 장에 HBM 메모리 수십 GB가 들어갑니다. SK하이닉스, Micron 수혜입니다. 넷째, 서버. GPU를 담을 서버 조립·공급이 필요합니다. DELL, HPE, SMCI가 여기에 해당합니다.

그리고 이 모든 것의 최종 소비자가 클라우드 빅3입니다. AWS, Google Cloud, Azure. 이들이 데이터센터 짓고, 반도체 사고, 메모리 구매하고, 서버 조립해서 AI 서비스를 팝니다.

데이터센터 투자 테마

데이터센터 관련 투자 테마를 네 가지로 정리했습니다.

첫 번째는 데이터센터 REIT입니다. EQIX(에퀴닉스), DLR(디지털 리얼티), AMT(아메리칸 타워). 땅과 건물을 임대해주는 회사들입니다. AI 서버 공간 수요가 폭증하면서 임대 가격이 오르고 있어요. 부동산처럼 배당도 줍니다.

두 번째는 전력 인프라입니다. VST, CEG, NRG. 데이터센터 하나가 쓰는 전력이 도시 하나 수준입니다. 2026년 미국 데이터센터 전력 소비가 미국 전체 전력의 4~8%가 될 거라는 예측이 있습니다. 발전소 운영 회사들이 조용히 뜨고 있는 이유입니다.

세 번째는 냉각 솔루션입니다. VRT(버티브), NVENT. H100 GPU는 전력 소비가 엄청나서 발열도 엄청납니다. 수냉식, 액침 냉각 등 특수 냉각 장비 수요가 폭발했습니다. 버티브는 2년 만에 주가가 10배 넘게 올랐습니다.

네 번째는 광통신·네트워크입니다. CIEN, ANET, CSCO. AI 추론 트래픽이 급증하면서 데이터센터 내부 네트워크, 데이터센터 간 연결망이 모두 업그레이드가 필요합니다. 조용한 수혜주입니다.

Slide 5-6
반도체 + 메모리 연결고리

엔비디아와 메모리 회사는 떼려야 뗄 수 없습니다. 이 관계를 이해하면 SK하이닉스와 Micron 투자 논리가 보입니다.

엔비디아 H100/B200 GPU에는 HBM3e라는 고대역폭 메모리가 80GB 탑재됩니다. AI 학습을 할 때 엄청난 양의 데이터를 빠르게 GPU 코어로 공급해야 하는데, 일반 DDR5 메모리로는 대역폭이 부족합니다. 그래서 HBM이라는 특수 메모리가 필요한 거예요.

HBM 시장을 SK하이닉스가 50% 이상 점유하고 있습니다. 엔비디아가 H100을 납품하면, 거기 들어가는 HBM도 SK하이닉스에서 삽니다. 그래서 엔비디아 실적 발표나 대규모 데이터센터 투자 뉴스가 나오면 SK하이닉스 주가도 같이 오르는 패턴이 반복됩니다.

미국 종목 중에는 Micron(MU)이 있습니다. 미국 내 유일한 DRAM 제조사입니다. HBM 시장에서는 후발주자지만 미국 정부 보조금과 공급망 다변화 수혜를 받고 있습니다. 애널리스트 EPS 전망이 2개월 만에 두 배가 된 적도 있습니다.

S&P 500이란?

S&P 500은 미국 주식 시장을 대표하는 지수입니다. 503개 종목으로 구성되어 있고, 전 세계 시가총액의 약 45%를 차지합니다. 코스피 전체가 전 세계의 2~3%인 것과 비교하면 규모가 다릅니다. 코스피가 아니라 삼성·SK·현대 최상위 503개를 모은 압축판 미국 경제라고 생각하시면 됩니다.

종목 비중은 시가총액 가중 방식입니다. SPY ETF 하나를 사면 이 503개 종목을 한 번에 보유하는 것과 같습니다. 지난 10년 수익률이 약 250%로, 같은 기간 코스피는 거의 제자리였습니다.

미국 주식의 또 다른 장점은 달러 자산이라는 점입니다. 위기가 올수록 안전자산 달러가 강세가 됩니다. 주가 상승 수익과 달러 헤지 효과를 동시에 누릴 수 있습니다.

시장 신호 3대장

시장 상태를 읽는 핵심 지표 3가지를 먼저 알아두십시오.

첫 번째는 VIX, 공포지수입니다. 20 이하면 시장이 평온한 상태, 30 이상이면 공포 상태, 80 이상이면 패닉입니다. 2020년 코로나 때 VIX가 85까지 올라갔습니다.

두 번째는 Yield Curve, 장단기 금리차입니다. 10년물 국채 금리에서 3개월물 금리를 뺀 값인데, 이 값이 마이너스로 역전되면 역사적으로 12~18개월 후에 경기침체가 왔습니다. 예외가 없었습니다.

세 번째는 HY Spread, 하이일드 스프레드입니다. HYG/IEF 비율로 측정하는데, 이 비율이 벌어지면 기업 부도 위험이 높아지고 있다는 신호입니다.

이 3가지가 바로 S06에서 배울 5센서 중 3개와 직결됩니다. VIX 센서(30%), Yield Curve 센서(12%), Credit 센서(15%)가 각각 이 지표를 사용합니다.

이 지표들이 나빠지면 us-stock이 자동으로 시장 체제를 바꿉니다.

ETF 지도

us-stock 시스템에서 사용하는 ETF들을 먼저 알아두겠습니다. SPY는 S&P 500을 추종하며 10년 수익률 약 250%입니다. QQQ는 나스닥100을 추종하며 10년 약 450%로 더 공격적입니다. RSP는 동일가중 S&P 500 ETF인데, 시장 폭(Breadth)을 측정하는 센서로 활용됩니다.

HYG는 하이일드 채권 ETF, IEF는 중기 국채 ETF입니다. 이 두 개의 비율이 Credit 센서가 됩니다. 그리고 섹터 ETF 11개가 있습니다. XLK(기술), XLF(금융), XLE(에너지), XLV(헬스케어), XLI(산업재) 등 11개가 S&P 500의 11개 섹터를 각각 대표합니다.

11개 섹터 ETF는 야구팀 포지션처럼 생각하시면 됩니다. 섹터 히트맵에서 초록이 많으면 시장이 건강하고, 빨간 섹터가 많으면 위험 신호입니다.

시장 체제란?

"오늘 주식 투자하기 좋은 날인가?" 이 질문에 감으로 대답하지 말고 숫자로 대답하는 것이 시장 체제 감지입니다. us-stock은 5센서 점수를 합산해서 4단계 체제를 판정합니다. risk_on(0.75 미만), neutral(0.75~1.50), risk_off(1.50~2.25), crisis(2.25 이상)입니다.

기상청의 날씨 예보처럼 생각하시면 됩니다. 맑음(risk_on), 흐림(neutral), 비(risk_off), 태풍(crisis) 4단계입니다. 태풍 예보가 떴는데 소풍을 가면 안 되듯, crisis 체제에서 신규 매수를 늘리는 것은 위험합니다.

중요한 것은 체제가 언제든 바뀔 수 있다는 점입니다. 그래서 매일 아침 5분 루틴으로 체제를 확인하는 습관이 필요합니다.

체제별로 손절 기준이 자동으로 바뀝니다.
Slide 5-12
이동평균선

이동평균선은 기술적 분석의 가장 기본 도구입니다. 5일, 20일, 60일, 120일, 200일 평균 주가를 선으로 연결한 것입니다. 단기선이 장기선 위에 차례로 쌓여 있는 상태를 정배열이라고 하며, 강한 상승 추세를 의미합니다.

골든크로스는 단기 이동평균선이 장기 이동평균선을 아래에서 위로 돌파하는 순간입니다. 대표적으로 50일선이 200일선을 위로 넘는 것이 골든크로스이며, 많은 시스템의 매수 신호가 됩니다. 반대로 위에서 아래로 내려오는 것이 데드크로스, 매도 신호입니다.

200일선은 지난 1년치 주가의 평균입니다. "200일선 위에서만 사라" — 간단하지만 강력한 필터입니다. us-stock은 SMA50과 SMA200을 Trend 센서에 활용합니다.

이동평균으로 추세를 봤다면 지금 과열인지는 RSI로 봅니다.
Slide 5-13
RSI · MACD · 볼린저밴드

RSI(Relative Strength Index)는 0에서 100 사이 숫자로 과매수·과매도를 측정합니다. 70 이상이면 과매수(조정 가능성), 30 이하면 과매도(반등 가능성)입니다. us-stock에서 RSI가 30 미만이면 Reversal(반전) 전략으로 분류됩니다.

MACD(Moving Average Convergence Divergence)는 두 개의 이동평균 차이를 이용해서 추세 전환을 감지합니다. MACD 선이 시그널 선을 위로 돌파하면 매수 신호, 아래로 내려오면 매도 신호입니다. 볼린저밴드는 이동평균선 ±2σ 구간을 표시하는데, 밴드가 수축되면 큰 변동성 움직임이 임박했다는 신호입니다.

이 3가지가 us-stock 6팩터 중 기술적 분석 25%의 원재료입니다. 단일 지표로 매매하면 오신호가 많지만, 여러 지표를 복합 점수로 만들면 신뢰도가 올라갑니다.

개별 종목 분석보다 시장이 그 종목을 어떻게 평가하는지가 더 중요합니다.
Slide 5-14
상대강도(RS)

상대강도(RS)는 종목 수익률에서 SPY 수익률을 뺀 값입니다. RS가 0보다 크면 그 종목이 시장 평균을 이기고 있다는 뜻입니다. RS가 0 미만이면 시장보다 못하고 있는 것입니다.

반 전체가 시험 볼 때 평균보다 높은 점수를 받는 학생처럼 생각하시면 됩니다. 기관은 언제나 반 평균보다 높은 성적표를 가진 학생(종목)만 사 모읍니다. 시장이 전체적으로 빠질 때도 상대적으로 덜 빠지는 종목이 RS 강한 종목입니다.

us-stock의 6팩터에서 RS는 15% 가중치를 갖습니다. RS > 0이고 이동평균 정배열이면 Trend 전략으로 분류됩니다. 가장 안정적인 추세 추종 방식입니다.

기관이 실제로 무엇을 사고 있는지는 13F 공시로 확인할 수 있습니다.
Slide 5-15
밸류에이션 핵심

주식 가격이 합리적인지 판단하는 핵심 도구는 PER입니다. 주가를 주당순이익(EPS)으로 나눈 값입니다. 투자는 미래를 사는 것이니 Forward PER(미래 이익 전망 기준)이 더 중요합니다.

현재 S&P 500 전체 Forward PER은 약 19.8배입니다. 역사적 평균이 16~17배이니 약간 높아 보이지만, AI 기업들의 이익 성장률을 감안하면 합리적인 범위입니다. Forward 12M EPS가 계속 올라가고 있다면, 주가가 올라도 PER이 낮아질 수 있습니다.

PER이 낮다고 무조건 좋은 것이 아닙니다. "싸 보이는 주식"이 실제로는 이익이 급감하고 있는 종목일 수 있습니다. ROE는 15% 이상이면 자본 효율이 우수한 기업입니다.

재무제표는 기업의 과거입니다. 이제 손실을 막는 법을 봅니다.
Slide 5-16
위험관리 — MDD와 Stop Loss

MDD(Maximum Drawdown) 비대칭성을 반드시 기억하십시오. 주식이 10% 떨어지면 원금 회복에 11.1% 상승이 필요합니다. 20% 손실은 25% 회복, 50% 손실은 100% 회복이 필요합니다. 손실이 커질수록 회복이 기하급수적으로 어려워집니다.

Stop Loss는 3단계입니다. 손실제한(-5% 자동 매도), 이익실현(+10% 일부 매도), Trailing Stop(최고가에서 일정 비율 하락 시 자동 매도). us-stock은 체제별로 기준이 자동으로 조정됩니다. risk_on: -10%, neutral: -8%, risk_off: -5%, crisis: -3%입니다.

Stop Loss는 안전벨트입니다. "첫 손절이 제일 손실이 작다" — 이 문장 하나만 오늘 가져가셔도 됩니다.

"손절 못 하고 들고 가다가 더 손해 본 경험 있으신 분? 그 종목이 결국 어떻게 됐습니까?"
기관 투자자가 어떻게 움직이는지 공개 데이터로 추적하는 방법이 있습니다.
Slide 5-17
13F — 스마트 머니 추적

13F는 미국 SEC의 의무 공시입니다. 운용자산 1억 달러 이상의 기관 투자자는 분기 종료 후 45일 이내에 보유 종목과 수량을 전부 공개해야 합니다. 이 데이터를 보면 워런 버핏의 버크셔 해서웨이, 르네상스 테크놀로지 같은 세계 최고 기관이 무엇을 사고 팔았는지 알 수 있습니다.

기관 지분이 증가하는 종목은 Strong Accumulation, 즉 대형 자금이 매집하고 있는 상태입니다. 반대로 기관 지분이 감소하면 Distribution, 분산 매도 상태입니다.

한 가지 중요한 원칙이 있습니다. Look-ahead bias 방지입니다. 13F는 분기가 끝나고 45일 후에 공개됩니다. us-stock은 공시일 기준으로만 데이터를 계산합니다. 공시 전의 데이터를 마치 알고 있는 것처럼 계산하면 백테스트 결과가 왜곡됩니다.

이 6가지 팩터가 합산되어 A~F 등급이 나옵니다.
Slide 5-18
Grade A~F + Action 예고

지금까지 배운 RSI·PER·RS·13F가 전부 us-stock의 6팩터 안에 들어있습니다. 기술적 분석(25%) + 펀더멘털(20%) + 애널리스트 의견(15%) + 상대강도(15%) + 거래량(15%) + 13F 기관지분(10%)을 합산하면 0~100점 복합 점수가 나옵니다.

80점 이상이면 Grade A(Strong Accumulation), 62점 이상이면 B, 48점 이상이면 C, 35점 이상이면 D, 35점 미만이면 F(Capitulation)입니다.

GO 체제에서 Grade A/B면 BUY, CAUTION 체제에서 A/B면 SMALL BUY, STOP 체제면 등급에 관계없이 HOLD입니다. 방금 배운 모든 개념이 이 하나의 매트릭스로 수렴됩니다.

왜 자동화가 필요한가

503개 종목에 6팩터를 매일 계산하면 어떻게 됩니까? 1종목당 수작업으로 10분씩 잡으면 503종목 × 10분 = 5,030분, 약 84시간입니다. 직장인이라면 불가능한 일입니다.

더 큰 문제는 인간의 편향입니다. FOMO(상승 놓칠까 봐 두려움), 손실회피 편향(이익보다 손실을 2배 더 고통스럽게 느낌), 확증 편향(내가 보유한 종목에 대한 긍정적 정보만 수집). 수식은 이런 편향이 없습니다. 매일 똑같은 기준으로 503개 종목을 동일하게 평가합니다.

완전 자동화로 이 문제를 해결합니다. 평일 아침 07:00에 macOS launchd가 자동으로 파이프라인을 실행합니다. 아침에 일어나면 결과가 나와 있습니다.

11 섹터 ETF 맵 & 로테이션

11개 SPDR 섹터 ETF입니다. XLK(기술), XLF(금융), XLE(에너지), XLV(헬스케어), XLI(산업재), XLY(경기소비재), XLP(필수소비재), XLU(유틸리티), XLRE(부동산), XLB(소재), XLC(통신)입니다.

섹터 로테이션은 경기 사이클에 따라 강세 섹터가 바뀌는 현상입니다. 경기 확장기에는 XLK·XLY가 강하고, 경기 후퇴기에는 XLP·XLU·XLV가 강합니다. 위기 국면에서는 XLP(생필품)가 방어주 역할을 합니다.

us-stock의 섹터 히트맵에서 이 11개 섹터의 현재 상태를 색상으로 표시합니다. 초록 섹터가 많을수록 시장이 건강하고, 빨간 섹터가 많을수록 위험 신호입니다.

Slide 5-21
기술 지표 3총사 — RSI·MACD·이동평균

이동평균(MA) — 추세 방향을 봅니다. 정배열이면 상승 추세, 역배열이면 하락 추세. 골든크로스는 매수, 데드크로스는 매도.

RSI — 과열·침체를 봅니다. 70 이상이면 과매수(팔 때), 30 이하면 과매도(살 때). 단, 강한 추세에서는 RSI가 70 이상 유지되기도 합니다.

MACD — 모멘텀 전환을 봅니다. MACD가 시그널 위로 올라오면 상승 모멘텀, 아래로 내려가면 하락 모멘텀.

이 3가지를 동시에 확인하면 오신호가 줄어듭니다. us-stock의 기술적 분석 25% 팩터가 이 3가지를 종합해서 점수를 냅니다.

Grade 판정 로직 흐름도를 보겠습니다.

Grade 판정 로직 흐름도

503개 종목이 어떻게 Grade를 받는지 흐름도로 정리합니다.

1단계: 데이터 수집. Yahoo Finance OHLCV + 재무 데이터 + 13F 공시 데이터.

2단계: 팩터 계산. 6팩터 각각을 0~100점으로 정규화합니다. 기술적(25%) + 펀더멘털(20%) + 애널리스트(15%) + RS(15%) + 거래량(15%) + 13F(10%).

Slide 5-23
Verdict × Action 매트릭스

Verdict는 3종류입니다. GO는 Regime 정상 + Gate GO + ML Bullish 세 조건이 모두 충족될 때입니다. STOP은 Regime이 crisis 또는 risk_off이거나 Gate가 STOP일 때입니다. CAUTION은 그 외 모든 경우입니다.

GO + Grade A/B = BUY. GO + Grade C 이하 = WATCH. CAUTION + Grade A/B = SMALL BUY. CAUTION + Grade C 이하 = WATCH. STOP + 전체 = HOLD.

중요한 원칙: 시장이 위기 국면(STOP)이면 아무리 Grade가 좋은 종목도 사지 않습니다. 좋은 종목도 시장 전체가 하락할 때 함께 내려갑니다.

Part 3 전략 × 셋업

Grade와 함께 전략(Strategy)과 셋업(Setup)이 분류됩니다.

전략 3종류입니다. Trend — RS > 0이고 MA 정배열. 추세 추종. 가장 안정적입니다. Swing — RSI 30~70 범위. 단기 반등을 노립니다. Reversal — RSI < 30이거나 데드크로스. 역발상 접근입니다.

셋업 3종류입니다. Breakout — 골든크로스이거나 RSI > 60. 돌파 매수입니다. Pullback — 강세이지만 RSI 40~55. 눌림 매수입니다. Base — 횡보 바닥 구간. 저점 매집입니다.

이 조합이 어떻게 쓰이냐면, "Trend 전략 + Breakout 셋업"이면 가장 공격적인 매수 시점입니다. "Reversal 전략 + Base 셋업"이면 역발상 저점 매수 시도입니다. 본인의 투자 성향에 맞는 조합을 고르십시오.

Slide 6-19
Part 4 AI 분석 노트 — 5섹션

AI가 종목마다 생성하는 분석 노트는 5섹션으로 구성됩니다.

1. Investment Thesis — 1~2문단으로 이 종목을 투자해야 하는 핵심 이유를 씁니다.

2. Bull Catalysts — 상승 촉매 3개. 앞으로 주가가 올라갈 구체적인 이유입니다.

3. Bear Risks — 하락 리스크 2~3개. 반드시 포함하도록 강제합니다. AI가 장밋빛 분석만 내놓는 것을 방지합니다.

4. Recommendation — 매수·매도·보유 중 하나. 이유와 함께 제시됩니다.

5. Key Data Points — 분석에 사용된 핵심 수치들. Forward PER, EPS 성장률, 애널리스트 목표가 등.

이 구조가 증권사 애널리스트 보고서와 유사합니다. 상승 촉매 2개 이상에 하락 리스크가 낮으면 긍정적입니다. AI Recommendation과 Grade가 일치하면 신뢰도가 높습니다.

Part 5 — SPY/QQQ 방향 예측

LightGBM 모델과 27개 피처로 SPY와 QQQ의 5일 방향(상승/하락)을 확률로 예측합니다. 6개 피처 카테고리가 있습니다. 가격·기술(RSI·MACD·MA), 추세·모멘텀, 거시·변동성(VIX·Fear&Greed), 신용·체제(HYG/IEF), 시장 폭(RSP/SPY), 앙상블(5/10/20일 가중)입니다.

ML 예측은 walk-forward 검증을 거쳤습니다. 미래 데이터를 사용하는 look-ahead bias 없이 검증합니다. 단, 예측이 100% 맞을 수는 없습니다. 5센서 체제 + Grade + ML 예측, 세 가지를 모두 확인한 뒤 최종 결정을 내리시기 바랍니다.

Part 5 Walk-forward 검증 방법을 설명드리겠습니다.
Slide 6-21
Part 5 — Walk-forward 검증

Walk-forward 검증은 ML 모델이 실제로 작동하는지 검증하는 방법입니다. 일반적인 백테스트의 문제점은 미래 데이터를 보고 과거 성과를 측정한다는 것입니다. 이를 Look-ahead bias라고 합니다.

Walk-forward는 이걸 방지합니다. 예를 들어 2년치 데이터가 있다면 첫 1년으로 훈련하고, 다음 3개월로 테스트합니다. 그 다음 3개월을 추가해서 다시 훈련하고, 그 다음 3개월로 테스트합니다. 이 과정을 반복합니다.

이렇게 하면 모델이 실제로 "과거에 없었던 데이터"로 테스트됩니다. 실제 운용 환경과 가장 유사한 검증 방식입니다. us-stock의 ML 모델은 이 방식으로 검증됩니다.

AP-01
Part 6 — VaR 상세

VaR(Value at Risk)는 주어진 신뢰도 수준에서 특정 기간 동안 발생할 수 있는 최대 손실을 추정합니다. us-stock은 95% 신뢰도의 1일 VaR를 사용합니다.

예를 들어 VaR가 -2%라면, 오늘 포트폴리오가 -2% 이상 하락할 확률이 5% 미만이라는 뜻입니다. 100일 중 5일은 -2% 이상 손실이 발생할 수 있습니다.

한계도 있습니다. VaR는 정상적인 시장 환경을 가정합니다. 2008년 같은 극단적 위기에서는 VaR가 과소 추정됩니다. 그래서 CDaR(Conditional Drawdown at Risk)와 함께 씁니다. CDaR는 VaR를 초과하는 손실의 평균입니다. 극단적 손실을 더 잘 잡아냅니다.

AP-02
Part 6 — Drawdown 분석

Drawdown은 고점에서 현재까지의 하락폭입니다. 포트폴리오가 최고점 100에서 현재 80이면 Drawdown은 -20%입니다.

/risk 페이지에서 표시하는 Drawdown 지표들입니다. Current Drawdown(현재 고점 대비 하락폭), Max Drawdown(전체 기간 최대 낙폭), Average Drawdown Duration(낙폭 지속 평균 기간), Recovery Time(낙폭에서 회복하는 데 걸린 평균 기간).

이 지표들을 보면 이 전략이 얼마나 고통스러운 구간을 얼마나 자주 겪는지 알 수 있습니다. Sharpe가 높아도 Max Drawdown이 -50%라면 그 구간을 버티기 매우 어렵습니다. 심리적으로 견딜 수 있는 Max Drawdown 범위 안에서 전략을 운용해야 합니다.

AP-03
Part 6 — 달걀 분산 감시

분산 투자의 원칙은 "달걀을 한 바구니에 담지 마라"입니다. us-stock의 집중도 감시가 이것을 자동으로 체크합니다.

집중도 경고 기준입니다. 단일 종목 비중이 20% 이상이면 MEDIUM 경고. 단일 섹터 비중이 40% 이상이면 MEDIUM 경고. 상관계수 0.8 이상의 종목이 3개 이상이면 HIGH 경고(사실상 같은 방향으로 움직이는 종목을 여러 개 가진 것).

이 경고가 발동되면 어떻게 해야 할까요? 일부 매도로 비중을 낮추거나, 다른 섹터의 종목을 추가해서 분산을 늘립니다. 상관계수가 낮은 종목끼리 조합하면 같은 위험에서 더 높은 수익을 낼 수 있습니다.

AP-04
Docker 컨테이너 구조

Docker는 애플리케이션을 컨테이너에 담아서 실행하는 기술입니다. Image는 불변의 설계도입니다. Container는 그 Image를 실행한 인스턴스입니다.

us-stock의 Docker 구조입니다. ghcr.io에 Next.js 대시보드 Image가 올라가있습니다. Synology NAS에서 이 Image를 pull해서 Container로 실행합니다. port 8889로 외부 접속을 받습니다.

코드가 바뀔 때는 GitHub Actions가 자동으로 처리합니다. git push → npm run build → Docker Image 생성 → GHCR 업로드 → Synology에서 docker pull. 이 전체 흐름이 자동입니다. 코드 한 줄 수정하면 5분 후에 NAS에 반영됩니다.

AP-05
데이터 배포 — Mac → NAS rsync

Mac에서 분석 결과를 Synology NAS로 전송하는 방법입니다. rsync를 씁니다.

rsync는 변경된 파일만 전송합니다. 처음에는 전체를 보내지만, 이후에는 변경된 것만 보냅니다. 매일 새로운 JSON 파일이 생기는 us-stock 구조에 적합합니다.

Synology NAS는 직접 SSH 접속이 안 되는 환경입니다. Tailscale을 통해 VPN으로 연결합니다. Mac에서 tailscale up 명령어로 접속하면 NAS가 사설 IP로 보입니다. 이 IP로 rsync를 실행합니다. 보안 원칙: API 키는 Mac에만 있습니다. NAS에 절대 복사하지 않습니다.

AP-06
코드 배포 — GHA + Tailscale + Docker

코드 변경이 자동으로 Synology NAS에 배포되는 흐름입니다.

1단계: git push → GitHub에 코드 올라갑니다. 2단계: GitHub Actions가 자동으로 실행됩니다. npm run build로 Next.js 프로덕션 빌드를 만들고, Docker Image를 빌드합니다. 3단계: ghcr.io(GitHub Container Registry)에 Image를 업로드합니다. 4단계: GitHub Actions가 Tailscale로 NAS에 접속합니다. 5단계: NAS에서 docker pull로 새 Image를 받고, 컨테이너를 재시작합니다.

이 전체 흐름이 git push 하나로 자동 실행됩니다. Claude Code가 코드를 수정하고 push하면 5분 후에 NAS 대시보드가 새로 뜨는 구조입니다.

AP-07
3가지 접근 경로

us-stock 대시보드에 접속하는 3가지 방법입니다.

1. localhost:3000 — Mac에서 npm run dev로 개발 서버를 켤 때입니다. 가장 최신 코드가 반영됩니다. 파이프라인 실행 직후 결과 확인에 씁니다.

2. NAS IP:8889 — 같은 WiFi 네트워크에 있을 때 NAS의 로컬 IP로 접속합니다. Docker 컨테이너가 실행 중이어야 합니다.

3. Tailscale IP:8889 — 외부에서 접속할 때입니다. Mac과 NAS 둘 다 Tailscale이 켜져 있어야 합니다. 출장 중이나 카페에서도 접속 가능합니다. VPN으로 암호화되어 안전합니다.

AP-08
SQLite data.db

us-stock은 두 가지 데이터베이스를 씁니다.

data.db — 주가 데이터와 분석 결과를 저장합니다. 503개 종목의 OHLCV, 6팩터 점수, Grade, AI 분석 결과가 여기 있습니다. 파이프라인이 실행될 때마다 업데이트됩니다.

board.db — 대시보드의 게시판 데이터입니다. 강의 관련 공지, 메모 등 사용자가 직접 입력하는 데이터를 저장합니다. NAS에서 쓰기 권한이 있는 유일한 DB입니다.

SQLite를 선택한 이유: 서버 없이 파일로 동작합니다. 백업이 파일 복사로 끝납니다. 개인 프로젝트 규모에 충분합니다. Claude Code로 데이터 조회가 쉽습니다. "data.db에서 Grade A 종목 모두 보여줘"라고 하면 바로 됩니다.

AP-18
왜 복붙이 실패하는가

Claude Code 초보자들이 가장 많이 하는 실수입니다. 유튜브나 블로그에서 Claude Code 프롬프트를 복사해서 붙여넣습니다. 그런데 결과가 다릅니다. 왜일까요?

이유는 컨텍스트입니다. 같은 프롬프트도 컨텍스트에 따라 완전히 다른 결과가 나옵니다. CLAUDE.md가 다르고, 프로젝트 구조가 다르고, 이전 대화 이력이 다르기 때문입니다.

해결책은 원리를 이해하는 것입니다. 프롬프트를 외우지 말고, 왜 그 프롬프트가 작동하는지 이해하십시오. 원리를 알면 자신의 상황에 맞게 변형할 수 있습니다. "따라치지 말고, 이해하세요" — 오늘 강의의 핵심 메시지입니다.

AP-19
AI 기억 = 컨텍스트

AI가 "기억"하는 방식을 이해하면 Claude Code를 훨씬 잘 쓸 수 있습니다.

AI는 대화 기록을 "컨텍스트 창"에 담아서 처리합니다. 컨텍스트 창에 들어있는 것만 알고, 나머지는 모릅니다. 대화가 길어지면 초반 내용이 밀려나기 시작합니다. 이것이 컨텍스트 한계입니다.

해결책은 CLAUDE.md와 Skill입니다. 중요한 내용을 파일로 저장해두면 AI가 언제든 참조할 수 있습니다. 컨텍스트 창에 없어도 됩니다.

또한 /clear 명령으로 컨텍스트를 초기화할 수 있습니다. 대화가 너무 길어졌거나 맥락이 꼬였을 때 씁니다. 새로운 작업을 시작할 때는 /clear로 깔끔하게 시작하는 것이 좋습니다.

AP-20
/clear와 /compact

Claude Code에서 자주 쓰는 두 가지 명령어입니다.

/clear — 현재 대화 컨텍스트를 완전히 초기화합니다. 새로운 작업을 시작할 때 씁니다. 이전 대화 내용이 전혀 없는 깨끗한 상태에서 시작합니다. CLAUDE.md는 여전히 로드됩니다.

/compact — 현재 컨텍스트를 요약해서 압축합니다. 대화가 너무 길어졌을 때 씁니다. 중요한 내용은 유지하되, 세부 내용을 요약해서 컨텍스트 공간을 확보합니다. /clear보다 부드러운 방식입니다.

언제 쓰느냐: 새 기능 작업을 시작할 때 → /clear. 대화가 길어져서 속도가 느려졌을 때 → /compact. 에러가 계속 반복될 때 → /clear 후 다시 시작.

AP-21
CLAUDE.md = AI 사규

CLAUDE.md를 잘 쓰는 것이 Claude Code를 잘 쓰는 핵심입니다.

좋은 CLAUDE.md의 구성요소입니다. 프로젝트 개요(이 프로젝트가 무엇을 하는지), 아키텍처 구조(어떤 파일이 어떤 역할인지), 실행 방법(어떻게 실행하는지), 금지 사항(하면 안 되는 것), 알려진 버그(이미 알고 있는 문제), 작업 규칙(파일 수정 전에 Read로 먼저 읽기 등).

CLAUDE.md가 잘 되어있을수록 Claude Code의 작업 품질이 올라갑니다. 신입 직원에게 사규를 잘 전달할수록 더 잘 일하는 것과 같습니다.

처음에는 비어있어도 됩니다. 작업하면서 "이건 꼭 알아야 해"라는 것들을 조금씩 추가해 나가십시오. 쌓일수록 강력해집니다.

AP-22
설계 먼저, 코딩 나중

Claude Code를 잘 쓰는 사람들의 공통점입니다. 구현을 요청하기 전에 설계를 먼저 합니다.

"Top Picks 페이지에 섹터 필터 추가해줘"보다 이게 낫습니다. "Top Picks 페이지에 섹터 필터를 추가하려고 해. 현재 구조를 파악하고, 어떻게 구현하면 좋을지 계획을 먼저 보여줘. 구현은 내가 승인한 다음에 해줘."

왜 이게 나을까요? AI가 일단 만들어버리면 마음에 안 들어도 되돌리기 어렵습니다. 계획을 먼저 보고 방향을 잡은 다음 구현하면 원하는 결과를 얻을 확률이 올라갑니다.

큰 작업일수록 더 중요합니다. "이 시스템 전체를 리팩토링해줘"가 아니라 "리팩토링 계획을 먼저 보여줘. 3단계로 나눠서."

AP-23
Karpathy LLM Wiki

Andrej Karpathy의 LLM Wiki 개념을 실제로 어떻게 적용하는지 설명드립니다.

3층 지식 관리 구조입니다. 1층(컨텍스트 창) — 지금 이 대화. 2층(CLAUDE.md + Skills) — 프로젝트 고정 지식. 3층(Obsidian + 코드베이스) — 축적된 전체 자산.

실용적 적용 방법입니다. 중요한 결정을 내리면 옵시디언에 기록합니다. 자주 쓰는 워크플로우는 Skill로 만듭니다. 프로젝트 규칙은 CLAUDE.md에 씁니다. 이렇게 하면 AI와 대화할 때마다 처음부터 설명하지 않아도 됩니다. 2층과 3층이 자동으로 로드됩니다.

이 구조를 갖추는 데 1~2개월이 걸립니다. 하지만 한 번 갖추고 나면 AI와의 협업 효율이 극적으로 올라갑니다.

AP-24
내비게이션과 운전자

Claude Code와의 관계를 비유로 설명드리겠습니다.

Claude Code는 내비게이션입니다. 운전자는 여러분입니다. 내비게이션이 아무리 좋아도 목적지를 정하는 건 운전자입니다. 내비게이션이 "좌회전하세요"라고 해도, 그 길이 막혀있으면 운전자가 판단해서 직진합니다.

Claude Code도 마찬가지입니다. 아무리 좋은 AI도 목표와 방향은 여러분이 정해야 합니다. "이 시스템으로 뭘 하고 싶은가?" — 이 질문의 답은 AI가 아닌 여러분이 가지고 있습니다.

투자도 마찬가지입니다. 시스템이 BUY 신호를 줘도, 최종 결정은 여러분입니다. AI는 분석을 도와주지만, 책임은 투자자 본인에게 있습니다.

AP-25
핵심 공식 레퍼런스

강의에서 나온 핵심 공식들을 정리합니다.

RSI = 100 - (100 / (1 + RS)), RS = 평균상승폭 / 평균하락폭 (14일 기준)

MACD = EMA12 - EMA26, Signal = MACD의 EMA9

Sharpe Ratio = (포트폴리오 수익률 - 무위험이자율) / 수익률 표준편차

VaR(95%) = 포트폴리오 가치 × 1.645 × 일일 변동성

MDD = (최저점 - 최고점) / 최고점 × 100

Alpha = 포트폴리오 수익률 - (무위험이자율 + 베타 × (시장 수익률 - 무위험이자율))

Forward PER = 현재 주가 / Forward EPS(12개월 예상 EPS)
